# v0.5 部署文档 — 横切能力

## 1. 环境与构建

与 [v0.2 部署文档](../v0.2/部署文档.md) 相同：JDK 17+、Maven；主配置 [application.yml](../../agent-jingu3-cs-server/src/main/resources/application.yml)。

**CI**：推送/PR 时 GitHub Actions 执行 `mvn test` 与 `mvn verify`（见仓库根目录 [`.github/workflows/ci.yml`](../../.github/workflows/ci.yml)）；亦可手动 **Run workflow**（`workflow_dispatch`）。

## 2. 数据源与迁移

- **开发默认**：H2 内存库 + **Flyway**（`classpath:db/migration`，脚本源目录见仓库 [`docs/data/migration`](../../docs/data/migration)）。启动时依次执行 `V1`～`V8`（含记忆表 V4/V5、身份/技能表 V6、用户提示词密文审计表 V7、事实元数据扩展 V8）。
- **生产**：覆盖 `spring.datasource.*` 为 MySQL 等持久库；**MyBatis-Plus** 与表结构以 **Flyway 迁移** 为准（勿用手工 `ddl-auto` 替代迁移）。保持 `spring.flyway.enabled=true`。
- **本地 MySQL / Redis / Milvus**：复制 [`application-local.yml.example`](../../agent-jingu3-cs-server/src/main/resources/application-local.yml.example) 为 `application-local.yml`（已 `.gitignore`），使用 `--spring.profiles.active=dev,local`。

## 3. 配置项摘要

| 键 | 说明 |
|----|------|
| `jingu3.cron.scheduler-enabled` | 是否轮询执行到期 Cron 任务 |
| `jingu3.cron.poll-interval-ms` | Cron 轮询间隔（毫秒） |
| `jingu3.engine.agent-team.max-specialist-rounds` | Agent Team 子 Agent LLM 轮数（之后还有一次合成调用） |
| `jingu3.user.id` / `jingu3.user.name` | 单用户种子；HITL `resolver_user_id` 等与此一致 |
| `jingu3.chat.persist-user-prompt` | 是否将**原始**用户输入以 AES-256-GCM 写入 `user_prompt_cipher`；默认 `false` |
| `jingu3.crypto.user-prompt-aes-key-base64` | 32 字节 AES 密钥的 Base64；**开启落库时必填**，勿提交仓库，建议环境变量注入（见 `application-local.yml.example`） |

## 4. 验收操作建议

1. **Cron**：`POST /api/v1/cron/tasks` 创建过去时间的 `nextRunAt`，观察日志或 DB 中 `last_status`。
2. **HITL**：`POST /api/v1/hitl/approvals` → `GET .../pending` → `POST .../approve` 或 `reject`。
3. **DST**：`PATCH /api/v1/dst/{conversationId}` 写入 JSON → `GET` 校验 → `POST .../confirm` 校验 `revision` 递增。
4. **Agent Team**：`POST /api/v1/chat`，body 中 `mode`=`AGENT_TEAM`，核对回复中含「合成答复」与子 Agent 轨迹（非流式）；流式核对 `specialist_1`…与 `synthesize` 步。
