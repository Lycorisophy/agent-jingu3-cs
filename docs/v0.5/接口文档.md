# v0.5 接口文档 — 横切 REST（`/api/v1`）

机器可读摘要见同目录 [openapi.yaml](./openapi.yaml)。

统一响应体为 `ApiResult<T>`：`success`、`code`、`message`、`data`、`timestamp`。下文仅列本版本新增或强相关的服务端路径。

## 1. Cron（已实现）

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/cron/tasks` | 创建定时任务（单次执行 MVP） |
| GET | `/api/v1/cron/tasks` | 列出当前配置用户下的任务 |

请求/响应字段见代码：`CreateScheduledTaskRequest`、`ScheduledTaskVo`。

## 2. HITL（已实现）

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/hitl/approvals` | 创建 `PENDING` 审批单 |
| GET | `/api/v1/hitl/pending?conversationId=` | 按会话列出待审批 |
| POST | `/api/v1/hitl/{id}/approve` | 批准 |
| POST | `/api/v1/hitl/{id}/reject` | 拒绝 |

请求体（创建）：`CreateHitlApprovalRequest` — `conversationId`、`runId`（可选）、`payloadJson`（字符串，常为 JSON 文本）。

## 3. DST / 对话状态（已实现，占位）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/v1/dst/{conversationId}` | 读取该会话状态；无记录时 404 |
| PATCH | `/api/v1/dst/{conversationId}` | 写入/更新 `state_json`，`revision`+1；可传 `expectedRevision` 乐观锁 |
| POST | `/api/v1/dst/{conversationId}/confirm` | 用户定稿占位：`revision`+1 |

## 4. 对话主链路（既有）

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/chat` | 同步对话；`mode=AGENT_TEAM` 时 Leader → 多轮 Specialist（`jingu3.tool.enabled=true` 时专员可走内置工具）→ 合成 |
| POST | `/api/v1/chat/stream` | SSE 流式 |

**客户端平台**：可选请求头 `X-Jingu3-Client-Platform`（如 `web`、`ios`），或请求体字段 `clientPlatform`；**请求头优先于 Body**。WebSocket 可在握手时带同名头，浏览器无法自定义握手头时可在首条 JSON 中传 `clientPlatform`。

**送入大模型的用户侧正文**：服务端在记忆增强（若开启）之后，会在正文前附加 **UTC 标准时间（ISO-8601）** 与 **平台标识**；**意图路由仍仅依据原始 `message` 字段**，无需客户端改写字面用户句。

**原始用户提示词落库（可选）**：配置 `jingu3.chat.persist-user-prompt=true` 且提供 `jingu3.crypto.user-prompt-aes-key-base64`（32 字节密钥的 Base64）时，将**未格式化**的原始 `message` 以 AES-256-GCM 写入表 `user_prompt_cipher`；未配置密钥时跳过落库并打日志。

Agent Team 行为受 `jingu3.engine.agent-team.max-specialist-rounds` 约束；流式中 `step` 标签含 `leader`、`specialist_{n}`、`synthesize`；专员轮若调用工具则另有 `TOOL_RESULT` 事件。提示词约束：专员不输出确认类卡片话术、不引导创建定时任务，由合成步（主协调）统一面向用户说明。
