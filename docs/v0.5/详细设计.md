# v0.5 详细设计 — Cron、HITL、State、Agent Team

## 1. 对照指南

| 指南章节 | 本设计落点 |
|----------|------------|
| §8 Cron | CronStore、定时触发、executeJob、通知（通知可简化为日志/Webhook 占位） |
| §10 HITL | 审批队列、审批 API、与会话恢复策略 |
| §9 State | 状态表、轨迹引用；小模型异步分析列为后续 |
| §7 Agent Team | 中心化多轮；消息模型 |

## 2. Cron

- **表字段（草案）**：`id`, `cron_expr`, `payload_json`, `enabled`, `next_run_at`, `last_run_at`。
- **触发**：调度线程扫描到期任务 → 投递执行器 → 更新 `last_run_at` / 计算 `next_run_at`。
- **并发**：单任务互斥（行锁或 `synchronized` key），防止重叠执行。

## 3. HITL

- **表字段（草案）**：`id`, `conversation_id`, `payload`, `status`, `created_at`, `decided_at`, `decision`。
- **API**：`POST /approve/{id}`, `POST /reject/{id}`, `GET /pending`（路径以最终实现为准）。
- **会话继续**：批准后由客户端携带 `approvalToken` 或服务端根据 `conversationId` 解锁——二选一，在实现章节定稿。

## 4. State

- **表字段（草案）**：与差距分析附录接近：`task_id`, `conversation_id`, `status`, `progress`, `result_json`, `error`, `updated_at`。
- **引擎**：`StateTrackingModeHandler` 改为读写存储；或新增 `StateService` 由 handler 调用。

## 5. Agent Team

- **数据结构**：`AgentMessage(role, content, stepIndex)` 列表；最后一步 `synthesize` 调用 LLM 生成用户可见回复。
- **范围控制**：`jingu3.engine.agent-team.max-specialist-rounds` 配置默认 2。

## 6. 依赖

- 数据源：H2/MySQL 择一，与 v0.6 记忆选型尽量一致以减少迁移。
