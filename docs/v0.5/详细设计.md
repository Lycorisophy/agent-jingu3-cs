# v0.5 详细设计 — Cron、HITL、State、Agent Team

## 1. 对照指南

| 指南章节 | 本设计落点 |
|----------|------------|
| §8 Cron | CronStore、定时触发、executeJob、通知（通知可简化为日志/Webhook 占位） |
| §10 HITL | 审批队列、审批 API、与会话恢复策略 |
| §9 State | 状态表、轨迹引用；小模型异步分析列为后续 |
| §7 Agent Team | 中心化多轮；消息模型 |

## 2. Cron

- **表字段（草案）**：`id`, `cron_expr`, `payload_json`, `enabled`, `next_run_at`, `last_run_at`。
- **触发**：调度线程扫描到期任务 → 投递执行器 → 更新 `last_run_at` / 计算 `next_run_at`。
- **并发**：单任务互斥（行锁或 `synchronized` key），防止重叠执行。

## 3. HITL

- **表（已实现）**：`hitl_approval`：`id`, `conversation_id`, `run_id`, `status`, `payload_json`, `created_at`, `resolved_at`, `resolver_user_id`。
- **API**：`/api/v1/hitl/approvals`（POST）、`/api/v1/hitl/pending`、`/api/v1/hitl/{id}/approve|reject`。
- **会话继续**：MVP 仅持久化与 REST；与 ReAct/工具挂起续跑、回调策略后续定稿。

## 4. State / DST

- **表（已实现）**：`dialogue_state`：`conversation_id` 唯一，`schema_version`, `state_json`, `revision`, `updated_at`。
- **API**：`/api/v1/dst/{conversationId}`（GET/PATCH）、`POST .../confirm`。
- **引擎**：`StateTrackingModeHandler` 仍为占位说明；与表联动可在后续版本接入。

## 5. Agent Team

- **行为（已实现）**：Leader 1 次 LLM → Specialist **N 轮**（`jingu3.engine.agent-team.max-specialist-rounds`，默认 2）→ **synthesize** 1 次 LLM；轨迹封装于 `AgentTeamResult`（多轮列表 + 合成文）。
- **流式**：`leader`、`specialist_{k}`、`synthesize` 各为独立 step + BLOCK。

## 6. 依赖

- 数据源：H2/MySQL 择一，与 v0.6 记忆选型尽量一致以减少迁移。
