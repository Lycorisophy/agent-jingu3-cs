## 1. 项目概述

### 1.1 项目背景
开发一款 **C/S 架构**的智能体（Agent）应用，服务端负责核心推理与工具调度，客户端提供跨平台（Windows/macOS/Linux）交互界面并承担本地资源访问职责。

### 1.2 设计目标
- **跨平台客户端**：一套代码，三端运行。
- **高性能服务端**：Java 生态，稳定可靠，易于扩展。
- **数据安全**：敏感文件不离开用户设备，通过客户端本地工具调用。
- **可扩展性**：支持多模型切换、插件化工具注册、分布式部署。
- **运维友好**：容器化部署，配置化切换组件。

### 1.3 核心功能
- 多轮对话与上下文记忆（短期+长期）
- 大语言模型（LLM）推理与流式输出
- 工具调用（网络搜索、文件读写、代码执行等）
- 本地文件系统安全访问（由客户端代理执行）

## 4. 模块详细设计

### 4.1 服务端模块

#### 4.1.1 网关层 (Gateway)

**功能要点**：
- **WebSocket 连接管理**：维护 `sessionId -> WebSocketSession` 映射；处理心跳（Ping-Pong）；断线重连时恢复会话上下文。
- **认证授权**：客户端连接时在 URL 参数或首条消息中携带 JWT Token，网关层校验并提取 `userId` 存入会话属性。
- **限流**：基于用户 ID 的令牌桶限流，防止恶意刷接口。
- **消息路由**：将不同类型的消息（聊天、工具响应）转发至对应的 `AgentEngine` 处理。

**技术实现**：
```java
@Configuration
@EnableWebSocket
public class WebSocketConfig implements WebSocketConfigurer {
    @Override
    public void registerWebSocketHandlers(WebSocketHandlerRegistry registry) {
        registry.addHandler(new AgentWebSocketHandler(), "/ws/agent")
                .setAllowedOrigins("*")
                .addInterceptors(new JwtHandshakeInterceptor());
    }
}
```

#### 4.1.2 智能体引擎 (Agent Engine)

核心是一个基于 **ReAct** 模式的循环控制器。

**状态机设计**：
```
用户输入 → [记忆检索] → [构建提示词] → [LLM 推理] 
    ↳ 产生文本回复 → 流式推送客户端 → 结束
    ↳ 产生工具调用 → [执行工具] → 观察结果 → 回到推理步骤
```

**记忆管理器**：
- **短期记忆**：使用缓存服务存储最近 N 轮对话历史（默认 10 轮）。Key 设计：`short_term:{conversationId}`。
- **长期记忆**：将历史对话内容向量化后存入 Qdrant，检索时基于当前用户输入进行相似度搜索，召回 Top-K 条相关记忆，注入提示词。

**提示词模板示例**：
```
你是一个智能助手，可以调用工具来完成任务。
当前时间：{{current_time}}
用户位置：{{user_location}}

可用工具：
{{tools_description}}

对话历史：
{{history}}

用户最新输入：{{user_input}}

请逐步思考并决定下一步行动。格式：
Thought: 你的思考过程
Action: 工具名称
Action Input: 工具参数 JSON
或者 Final Answer: 最终回复给用户的内容
```

#### 4.1.3 工具调用调度器

**工具分类**：
- **服务端工具**：无需客户端参与，直接在服务端执行（如网络搜索、数据库查询、调用第三方 API）。
- **客户端工具**：需要访问用户本地资源（如读写文件、执行命令），由服务端通过 WebSocket 下发指令，客户端本地代理执行后回传结果。

**工具接口定义（本仓库）**：
```java
/** 内置可调用工具契约：由 ToolRegistry 统一注册与执行（包 cn.lysoy.jingu3.tool） */
public interface Jingu3Tool {
    String id();
    String description();
    String execute(String input) throws cn.lysoy.jingu3.tool.ToolExecutionException;
}
```
（历史草案中的 `AgentTool` / JSON Schema 输入输出可作为客户端工具或后续企业化扩展的参考形态，与当前 `Jingu3Tool` 字符串契约并存演进。）

**工具调用流程（客户端工具示例）**：
1. LLM 决定调用 `read_file`，服务端生成唯一 `toolCallId`。
2. 服务端通过 WebSocket 发送：
   ```json
   {
     "type": "tool_call",
     "toolCallId": "uuid-1234",
     "toolName": "read_file",
     "params": { "path": "/home/user/report.txt" }
   }
   ```
3. 客户端本地代理调用 Node.js `fs.readFile`，获取内容。
4. 客户端回传：
   ```json
   {
     "type": "tool_result",
     "toolCallId": "uuid-1234",
     "result": { "content": "文件内容..." }
   }
   ```
5. 服务端将结果作为观察（Observation）输入 LLM，继续推理。

#### 4.1.4 LLM 适配器

- **接口抽象**：支持 OpenAI、Azure OpenAI、Ollama、通义千问等多种模型。
- **流式输出**：使用 Reactive Streams 或 `Flux<String>` 将模型生成的 Token 逐块推送至客户端。
- **容错处理**：失败重试、降级（如主模型不可用时切换备用模型）。

#### 4.1.5 缓存服务（可插拔设计）

**接口定义**：
```java
public interface CacheService {
    void set(String key, Object value, long ttlSeconds);
    <T> T get(String key, Class<T> clazz);
    void delete(String key);
    boolean exists(String key);
}
```

**实现切换机制**：
通过 Spring Boot 的 `@ConditionalOnProperty` 注解，根据配置文件 `cache.type` 的值（`local` 或 `redis`）动态装载对应的 Bean。

- **本地实现**：基于 Caffeine，适合开发与单机部署。
- **Redis 实现**：基于 Spring Data Redis，支持分布式环境共享缓存。

**配置示例** (`application.yml`)：
```yaml
cache:
  type: redis   # 或 local

# 当 type=redis 时生效
spring:
  data:
    redis:
      host: ${REDIS_HOST:localhost}
      port: ${REDIS_PORT:6379}
```

### 4.2 客户端模块

#### 4.2.1 UI 视图层

- **技术**：React 18 + TypeScript + Ant Design / Tailwind CSS。
- **核心组件**：
  - **聊天窗口**：支持 Markdown 渲染、代码高亮、流式文字效果。
  - **工具调用卡片**：折叠展示 Agent 的思考过程（Thought）和工具调用参数/结果。
  - **设置面板**：模型选择、本地目录授权管理、网络代理配置。

#### 4.2.2 本地代理层

- **WebSocket 客户端**：负责连接保活、自动重连、消息序列化/反序列化。
- **指令执行器**：监听服务端下发的 `tool_call` 消息，根据 `toolName` 分发到对应的本地工具实现。
- **本地工具集示例**：
  - `read_file`：读取指定路径文件内容。
  - `write_file`：写入内容到指定路径（需用户确认）。
  - `list_directory`：列出目录内容。
  - `execute_command`：执行 Shell 命令（高风险，需用户授权）。

#### 4.2.3 通信安全

- **TLS 加密**：所有 WebSocket (`wss://`) 和 HTTP (`https://`) 通信强制加密。
- **Token 存储**：JWT 存储在系统安全存储区（如 macOS Keychain、Windows Credential Manager）。

#### 4.3.2 向量数据库设计参考（Qdrant）

**Collection 定义**：

```json
{
  "name": "agent_memory",
  "vectors": {
    "size": 1536,
    "distance": "Cosine"
  }
}
```

**Payload 结构**（每条向量点的附加信息）：
```json
{
  "user_id": "uuid",
  "conversation_id": "uuid",
  "message_id": "uuid",
  "role": "user/assistant",
  "content_snippet": "前200字摘要",
  "timestamp": "2026-04-12T10:00:00Z"
}
```

**检索流程**：
1. 用户输入文本 → 调用 Embedding 模型生成向量。
2. 使用向量在 Qdrant 中搜索，同时添加过滤条件 `user_id = ?` 保证数据隔离。
3. 获取 Top K 结果，根据 `message_id` 关联 MySQL 获取完整内容，注入提示词。

## 5. 非功能需求设计

### 5.1 安全性
- **传输安全**：全链路 TLS 1.3。
- **认证授权**：JWT Token 有效期 24 小时，敏感操作（如删除会话）需验证 Token 中的用户身份。
- **数据隔离**：所有数据库查询、向量检索必须带上 `user_id` 条件。
- **工具调用安全**：
  - 客户端本地工具调用需用户授权（弹窗确认）。
  - 服务端工具参数进行严格校验，防止注入攻击。
  - 高风险工具（如执行命令）默认禁用，需在配置中显式开启。

### 5.2 性能
- **WebSocket 连接数**：单机支持 10,000 并发连接（通过 Netty 或 WebFlux 实现）。
- **流式响应延迟**：首 Token 延迟 < 1s（不含 LLM 推理时间）。
- **缓存策略**：用户会话信息、短期记忆缓存命中率 > 95%。
- **数据库优化**：对高频查询字段建立索引；向量检索结果缓存 5 分钟。

### 5.3 可扩展性
- **服务端水平扩展**：通过 Nginx 或 Kubernetes Ingress 负载均衡 WebSocket 连接，会话粘滞（Sticky Session）或使用 Redis 共享会话状态。
- **向量库扩展**：Qdrant 支持分布式部署，可按 `user_id` 哈希分片。
- **插件化工具**：新增内置工具实现 `Jingu3Tool` 并注册为 Spring Bean，由 `ToolRegistry` 收集，无需修改注册表核心代码。

### 5.4 可观测性
- **日志**：结构化日志（JSON 格式），包含 `traceId`、`userId`，便于链路追踪。
- **指标**：通过 Micrometer + Prometheus 暴露指标，监控 QPS、WebSocket 连接数、工具调用成功率。
- **健康检查**：提供 `/actuator/health` 端点，包含 MySQL、Redis、Qdrant 连通性检测。

## 8. 总结

本文档从架构、技术选型、模块设计、数据存储、通信协议、安全、部署等多个维度，完整阐述了跨平台 CS 架构智能体系统的设计方案。核心亮点包括：

1. **跨平台客户端**采用 Electron，一套代码覆盖主流桌面系统。
2. **服务端**基于 Spring Boot + LangChain4j，稳定且易于集成 AI 能力。
3. **数据库**使用 MySQL 存储结构化数据，配合独立向量数据库 Qdrant 实现高效长期记忆检索。
4. **缓存**设计为可插拔接口，支持本地 Caffeine 与 Redis 无缝切换，适应不同部署规模。
5. **客户端本地代理**机制保障了用户文件安全，同时扩展了智能体的本地操作能力。

该设计已充分考虑生产环境的高可用、可扩展与可观测性要求，可直接作为开发蓝图。如有具体模块需深化设计（如详细类图、API 文档示例、部署脚本等），可进一步沟通。
