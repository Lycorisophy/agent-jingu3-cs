## 2. 核心类型

### 2.1 `ActionMode`（枚举）

八类：`ASK`, `REACT`, `PLAN_AND_EXECUTE`, `WORKFLOW`, `AGENT_TEAM`, `CRON`, `STATE_TRACKING`, `HUMAN_IN_LOOP`。

### 2.2 `RoutingSource`

`CLIENT_EXPLICIT`, `RULE`, `MODEL`。

### 2.3 `IntentRouter.resolve(ChatRequest)`

1. 若 `request.mode != null`：返回 `CLIENT_EXPLICIT` + 该模式。
2. 否则规则表：关键词子串匹配（不区分大小写可选），命中则 `RULE`。
3. 否则 `ModelIntentClassifier.classify(userMessage)`：`MODEL`；异常或无法解析 → `REACT` + 记录 warn。

### 2.4 `ModeRegistry`

启动时注册：Ask、ReAct、Stub（同一 Stub 实例可注册多枚举或每枚举一个 Stub）。

### 2.5 `ExecutionContext`

字段示例：`userId`, `username`, `conversationId`, `userMessage`, `selectedMode`, `routingSource`, `history`（`List<String>`，v0.1 可空或单轮）。

### 2.6 Ask / ReAct

- **Ask**：系统提示（角色 + 约束）+ 用户消息 → `chatLanguageModel.generate(...)`。
- **ReAct**：系统提示要求「思考—行动或最终答案」，限制最大轮次（如 1～2 次 generate 简化）；v0.1 可不接真实工具，以文本观测为主。

## 3. REST API

见同目录 [接口文档.md](./接口文档.md)。

## 4. 日志

结构化 INFO：`userId`, `routingSource`, `actionMode`, `conversationId`（可选）。

## 5. 错误处理

- 未知 `mode` 字符串：400 + 可接受枚举列表。
- Ollama 不可用：500 + 明确错误信息（不打印密钥）。
