# v0.4 详细设计 — Plan + Workflow

## 1. 对照指南

| 指南章节 | 本设计落点 |
|----------|------------|
| §5 Plan-and-Execute | Executor 经 `ToolStepService` 与 `ToolRegistry` 集成；Replanner 逻辑保持 |
| §6 Workflow | 节点类型；顺序 LLM / TOOL 链 |

## 2. `ToolStepService`（`engine.support`）

- **职责**：封装「可选工具路由 → 执行 → 汇总」与「工作流 TOOL 节点执行」。
- **依赖**：`ChatLanguageModel`、`PromptAssembly`、`ToolRegistry`、`Jingu3Properties`、`ObjectMapper`。
- **Plan 子任务**：若 `jingu3.tool.enabled` 为 false，直接 `buildSubtaskExecutePrompt` + `generate`。否则先 `buildPlanSubtaskToolRouterPrompt`，`ToolRoutingParser.parseAskRoute`（与 Ask 同 JSON），`direct` 则同上；`tool` 则 `registry.execute`，可选 `StreamEventSink#toolResult`，再 `buildPlanSubtaskAfterToolPrompt` 或失败用 `buildPlanSubtaskToolFailurePrompt`。
- **Workflow TOOL 节点**：`toolId` 必填；`instruction` 为工具 `input` 字符串；成功将 `[工具 x 输出]\n...` 拼入 `accumulated`；失败拼错误句；流式时 `toolResult`。

## 3. Plan 与 Replanner

- 子任务内工具失败：**不抛** `ToolExecutionException` 到外层，由失败汇总提示词生成文本，**不触发** Replanner（与「工具可恢复」一致）。
- 子任务 `generate` 抛 `RuntimeException`：行为与现网一致，可触发 Replanner。

## 4. Workflow 节点模型

### 4.1 JSON 示例

```json
{
  "id": "demo-wf-tool",
  "nodes": [
    { "id": "n1", "type": "LLM", "instruction": "用一句话概括用户需求。" },
    { "id": "n2", "type": "TOOL", "toolId": "calculator", "instruction": "2+3" },
    { "id": "n3", "type": "LLM", "instruction": "根据上文工具输出，用中文给出最终结论。" }
  ]
}
```

- `type` 缺省或空 → **LLM**。
- **TOOL**：`toolId` 必填；`instruction` → 工具单参数（如表达式、或空串给 `utc_time`）。

### 4.2 执行顺序

- 数组顺序；`accumulated`：每步输出（LLM 生成全文或 TOOL 拼接段）作为下一步的「当前上下文与中间结果」输入。

## 5. 流式事件

- Plan：`STEP_BEGIN/END` + `BLOCK` 每子任务不变；若该子任务走工具，在对应 `BLOCK` 前增加 **`TOOL_RESULT`**（与 v0.3 一致）。
- Workflow：每节点一步；TOOL 节点可发 `TOOL_RESULT` 后 `BLOCK` 带更新后的 `accumulated`（或与 LLM 节点一致仅 `BLOCK` 全文，实现取后者便于前端统一）。

## 6. 测试

- `WorkflowNode` 反序列化：缺省 `type`、TOOL 节点。
- `ToolStepService`：Mock `ChatLanguageModel` 固定路由 JSON，断言 `ToolRegistry.execute` 调用。
- `WorkflowModeHandler`：可 Mock 注册表 + ChatModel 做 TOOL 链冒烟（可选）。
