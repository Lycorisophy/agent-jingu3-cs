## 1. B1 显式模式守门（已实现要点）

### 1.1 行为

- 客户端显式 `mode` 为 `PLAN_AND_EXECUTE` 或 `AGENT_TEAM` 时，在进入 Handler 前调用 `ModelIntentClassifier#classifyOptional`（一次短 LLM）。
- 若分类结果为 **ASK** 或 **REACT**：实际路由为 **ASK**，`routingSource` 为 **`EXPLICIT_GUARD`**，`RoutingDecision#guardUserNotice` 携带用户可见说明（见 `EngineMessages.EXPLICIT_MODE_GUARD_NOTICE`）。
- 若分类器 **调用失败**（`classifyOptional` 返回 empty）：**不降级**，尊重用户显式模式。
- 配置 **`jingu3.routing.explicit-mode-guard-enabled`**（默认 `true`）：设为 `false` 时**不调用** `classifyOptional`，显式 Plan/Team **直通** Handler（与关闭守门前的 v0.4 行为一致）。
- **ReAct / Workflow** 的显式选择：**当前不与** Plan/Team **同等守门**（阶段 C 决议见下）。

### 1.2 流式约定（SSE / WebSocket）

事件顺序：

1. **META**：`actionMode` = 实际执行模式（降级时为 `ASK`），`routingSource` = `EXPLICIT_GUARD`（若发生守门）。
2. **BLOCK**（仅当守门）：整段 `guardUserNotice`，与正文分离；客户端可渲染为提示条或折叠区。
3. **TOKEN**（Ask 流式）或后续 **STEP_*** / **BLOCK**（其它模式）：照常。

**用户可见性**：说明与正文属于**同一次用户提问的同一轮回复**；通过 **独立 BLOCK 帧** 与后续 TOKEN 区分，前端可选择「一条气泡内上下两段」或「系统提示 + 正文」样式。

### 1.3 非流式 `/chat`

`reply` = `guardUserNotice` + `execute(Ask)` 结果字符串拼接（无前缀重复）。

## 2. 阶段 C 短决议（产品/协议）

| 议题 | 决议（当前版本） |
|------|------------------|
| ReAct、Workflow 是否同等守门？ | **否**。仅 **Plan-and-Execute**、**Agent Team** 启用守门；后续若需统一，增加配置开关 `jingu3.routing.guard-explicit-modes` 扩展枚举即可。 |
| DST 定稿卡 vs HITL 审批卡 | **协议层分离**：流式扩展字段建议统一使用 `cardKind`（或顶层 `type`）区分：`state_digest`（状态核对/定稿，可编辑字段快照）与 `action_approval`（行动授权，含 toolId/风险等级）。**禁止**混用同一 `type` 值；客户端按 `cardKind` 挂载不同 UI。 |
| 命名（建议） | 事件名占位：`STREAM_CARD` 或复用 **BLOCK** + JSON `block` 内嵌结构化体；定稿前与前端共审一份 OpenAPI / JSON Schema。 |

| 项 | 草案 |
|----|------|
| 表 | `hitl_approval`：`id`, `conversation_id`, `run_id`, `status`（PENDING/APPROVED/REJECTED）, `payload_json`（摘要+原始 tool 参数）, `created_at`, `resolved_at`, `resolver_user_id` |
| REST | `POST /api/v1/hitl/{id}/approve`、`POST /api/v1/hitl/{id}/reject`；`GET /api/v1/hitl/pending?conversationId=`；`POST /api/v1/hitl/approvals` 创建 pending |

| 流式 | 挂起前发 **BLOCK** 或专用 **CARD**（`cardKind=action_approval`）；恢复后同 `run_id` 续跑 |

## 5. B4 Cron（极简）

| 项 | 草案 |
|----|------|
| 表 | `scheduled_task`：`id`, `owner_user_id`, `scope`（GLOBAL / CONVERSATION）, `conversation_id`（可空）, `cron_expr` 或 `next_run_at`, `payload_json`（用户意图/系统提示模板）, `enabled`, `last_run_at`, `last_status` |
| 创建 | （1）管理 UI / `POST /api/v1/cron/tasks`（与现有 `/api/v1` 前缀一致）；（2）工具 `create_scheduled_task` 写库（待办） |
| 调度 | Spring `@Scheduled` 轮询 `next_run_at <= now` 且 `enabled`；执行器内组装 `ChatRequest`（注入 `conversationId` 若会话级）经 `CronChatBridge` 调 `ChatService`；**MVP**：执行后 `enabled=false`，`cron_expression` 仅占位 |
| 到期后「起对话」 | **会话级**：固定 `conversationId`，用户可见为同一会话续写；**全局**：可用 `conversationId = "cron:{taskId}:{runInstance}"` 或系统会话，产品决定是否在 UI 展示 |
