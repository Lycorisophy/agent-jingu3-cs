# AI智能体行动模式实现差距分析报告

> 生成时间：2026-04-12
> 分析版本：v0.2
> 维护者：Matrix Agent

---

## 一、概述

本文档对 jingu3 项目中 **8种行动模式**的实现与设计指南进行深度对比分析，找出差距并给出改进建议。

**核心结论**：当前实现与设计指南存在显著差距，核心问题是 **缺少工具系统（Tool Registry）**，导致 ASK/REACT 等模式无法真正工作。

---

## 二、对照表总览

| 模式 | 设计复杂度 | 实现复杂度 | 实现质量 | 差距说明 |
|:----:|:----------:|:----------:|:--------:|----------|
| **ASK** | ⭐ | ⭐ | ❌ 差 | 未实现工具路由，只有简单LLM调用 |
| **REACT** | ⭐⭐⭐ | ⭐⭐ | ❌ 差 | 伪循环，没有真正工具调用和观察机制 |
| **PLAN** | ⭐⭐⭐⭐ | ⭐⭐⭐ | ✅ 较好 | 有Planner/Executor/Replanner，结构完整 |
| **WORKFLOW** | ⭐⭐⭐ | ⭐⭐ | ⚠️ 中 | 有节点概念，但fallback为简单两步 |
| **AGENT_TEAM** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⚠️ 中 | Leader+多轮 Specialist+合成；`jingu3.tool.enabled` 时专员复用 Ask 工具路由与主/子提示词边界；仍无多进程/消息总线级真多 Agent |
| **CRON** | ⭐⭐⭐ | ⭐ | ❌ 差 | 无真实调度器，只是演示文本 |
| **STATE** | ⭐⭐⭐⭐ | ⭐ | ❌ 差 | 只有计数器，无状态表、无小模型分析 |
| **HITL** | ⭐⭐⭐ | ⭐ | ❌ 差 | 无审批队列，只是返回提示 |

## 三、详细差距分析

### 3.1 ASK 模式 - 问题最严重

#### 设计指南要求（§3）

```
用户输入 → Agent接收 → 工具选择器 → 查询工具 → 返回结果
```

**核心组件**：
- Tool Registry（工具注册表）
- 工具选择器/路由器
- 多种查询工具（Search/DB/Knowledge）

#### 实际实现

**代码位置**：`engine/mode/AskModeHandler.java`

```java
@Override
public String execute(ExecutionContext context) {
    String combined = prompts.buildAskCombinedPrompt(context.llmInput());
    return chat.generate(combined);  // ❌ 只是简单调用LLM！
}
```

#### 问题清单

| 问题 | 严重程度 | 说明 |
|------|:--------:|------|
| 无Tool Registry | 🔴 P0 | 没有工具注册和管理机制 |
| 无工具路由器 | 🔴 P0 | 无法根据用户意图选择工具 |
| 纯LLM生成 | 🔴 P0 | 没有任何工具调用 |

#### 设计预期 vs 实际实现

```
设计预期：
┌──────────┐    ┌───────────┐    ┌─────────┐    ┌──────────┐
│ 用户输入 │ → │ 意图识别  │ → │ 路由   │ → │ 调用工具 │
└──────────┘    └───────────┘    └────┬────┘    └────┬─────┘
                                      │              │
                              ┌───────┴───────┐      │
                              │   工具列表     │      │
                              │ Search/Doc/DB │ ←────┘
                              └───────────────┘

实际实现：
┌──────────┐    ┌───────────┐    ┌─────────┐
│ 用户输入 │ → │ 拼接Prompt │ → │ 调用LLM │
└──────────┘    └───────────┘    └─────────┘
```

### 3.2 REACT 模式 - 伪循环

#### 设计指南要求（§4）

```
思考(Thought) → 行动(Action) → 观察(Observation) → 思考 → ... → 完成
```

**核心机制**：
- 真正的工具调用（Action）
- 工具执行结果反馈（Observation）
- 动态决策是否继续循环

#### 实际实现

**代码位置**：`engine/mode/ReActModeHandler.java`

```java
@Override
public String execute(ExecutionContext context) {
    String userMessage = context.llmInput();
    StringBuilder trace = new StringBuilder();
    for (int step = 1; step <= maxSteps; step++) {
        String stepPrompt = prompts.buildReactLoopStepPrompt(userMessage, prior, step, maxSteps);
        String out = chat.generate(stepPrompt);  // ❌ 只是反复调用LLM！
        trace.append(out);
    }
    return "【ReAct 轨迹】\n" + trace;
}
```

#### 问题清单

| 问题 | 严重程度 | 说明 |
|------|:--------:|------|
| 无真正Action | 🔴 P0 | 没有调用任何工具 |
| 无真正Observation | 🔴 P0 | 没有执行结果反馈 |
| 上下文无变化 | 🔴 P0 | 循环只是重复调用LLM |
| 提示词玩文字游戏 | 🟡 P1 | 让LLM自己写"思考/行动/观察" |

#### 设计预期 vs 实际实现

```

┌──────────┐
│  思考    │  "我需要先搜索相关信息"
│ Thought  │
└────┬─────┘
     │
     ▼
┌──────────┐     ┌─────────────────────────────────┐

│  Action  │     │  ┌─────────┬─────────┬─────────┐│
│          │     │  │ Search  │   API   │  File   ││
│          │     │  │  Tool   │  Tool   │  Tool   ││

     │                        ▼
```

### 3.3 PLAN 模式 - 相对最完整

#### 设计指南要求（§5）

```
规划阶段 → 执行阶段 → 结果整合 → [失败时重规划]
```

**核心组件**：
- Planner（规划器）
- Executor（执行器）
- Replanner（重规划器）

#### 实际实现

**代码位置**：`engine/mode/PlanAndExecuteModeHandler.java`

```java
@Override
public String execute(ExecutionContext context) {
    // ✅ 规划阶段
    String plan = chat.generate(prompts.buildPlanAndExecutePlanPrompt(in));
    return executeWithPlan(plan, context.getUserMessage(), false);
}

private String executeWithPlan(String plan, String originalUserMessage, boolean afterReplan) {
    // ✅ 解析子任务
    List<String> subtasks = PlanTextParser.parseSubtasks(plan);
    
    List<String> stepOutputs = new ArrayList<>();
    try {
        for (int i = 0; i < subtasks.size(); i++) {
            // ✅ 执行阶段（Executor）
            String r = chat.generate(
                prompts.buildSubtaskExecutePrompt(subtasks.get(i), originalUserMessage, i + 1));
            stepOutputs.add(r);
        }
    } catch (RuntimeException ex) {
        // ✅ 重规划
        if (replanEnabled && !afterReplan) {
            String newPlan = chat.generate(
                prompts.buildReplannerPrompt(plan, ex.getMessage(), originalUserMessage));
            return executeWithPlan(newPlan, originalUserMessage, true);
        }
        throw ex;
    }
    return 组装结果(plan, stepOutputs);
}
```

#### 评价

| 方面 | 状态 | 说明 |
|------|:----:|------|
| 规划阶段 | ✅ | 有独立的Planner提示词 |
| 执行阶段 | ⚠️ | Executor仍是LLM生成，非真实工具调用 |
| 重规划 | ✅ | 有失败重试机制 |
| 计划解析 | ✅ | 有PlanTextParser解析子任务 |

### 3.4 WORKFLOW 模式 - 有概念但简化

#### 设计指南要求（§6）

```
节点定义 → 工作流引擎 → 状态管理 → 错误处理
```

**四大工作流模式**：
- Prompt Chaining（顺序执行）
- Routing（路由）
- Parallelization（并行化）
- Evaluator-Optimizer（评估优化）

#### 实际实现

**代码位置**：`engine/mode/WorkflowModeHandler.java`

```java
@Override
public String execute(ExecutionContext context) {
    WorkflowDefinition def = workflowDefinitionRegistry.get(wf);
    if (def == null || def.getNodes() == null || def.getNodes().isEmpty()) {
        log.warn("workflow definition not found, fallback two-step");
        return fallbackTwoStep(in, context.getUserMessage());  // ⚠️ 回退为两步
    }

    String accumulated = in;
    for (WorkflowNode node : def.getNodes()) {
        accumulated = chat.generate(
            prompts.buildWorkflowNodePrompt(node.getInstruction(), accumulated, ...));
    }
    return accumulated;
}
```

#### 问题清单

| 问题 | 严重程度 | 说明 |
|------|:--------:|------|
| 无真正节点类型 | 🟡 P1 | 只有LLM节点，无Tool/Aggregator节点 |
| fallback过于简单 | 🟡 P1 | 两步回退只是Summary+Answer |
| 无并行执行 | 🟡 P1 | 只有顺序执行 |
| 无条件分支 | 🟡 P1 | 没有Router概念 |

## 四、根本原因分析

### 4.1 核心依赖缺失

| 依赖项 | 影响模式 | 说明 |
|--------|----------|------|
| **Tool Registry** | ASK, REACT | 所有需要工具调用的模式都无法工作 |
| **Agent通信机制** | AGENT_TEAM | 无独立 Agent 总线/并行 worker；专员与 Ask 共享工具链，非分布式协作 |
| **调度器** | CRON | 无法实现定时任务 |
| **状态存储** | STATE, HITL | 无法持久化状态 |

### 4.2 架构问题

```
当前架构：
┌──────────────┐
│  ActionMode  │ ← 各模式直接调用 chat.generate()
│   Handler    │
└──────┬───────┘
       │ 依赖链断裂
       ▼
┌──────────────┐
│   LLM Only   │ ← 没有工具系统
└──────────────┘

设计架构：
┌──────────────┐
│  ActionMode  │
│   Handler   │
└──────┬───────┘
       │
       ▼
┌──────────────┐    ┌──────────────┐
│  Tool        │ ←→ │    LLM       │
│  Registry    │    │  (决策中心)  │

       ▼
```

#### Phase 4: AGENT TEAM（v0.6）

**目标**：实现真正的多Agent协作

**架构设计**：
```java
// Agent定义
public interface Agent {
    String getName();
    String getRole();
    String process(String task, AgentContext context);
    void receive(AgentMessage message);
    void send(AgentMessage message);
}

// 团队Leader
public class TeamLeader implements Agent {
    private List<Agent> specialists;
    private Map<String, String> taskAssignments;
    
    public String process(String task, AgentContext context) {
        // 1. 分解任务
        List<SubTask> subtasks = decompose(task);
        
        // 2. 分配给专家Agent
        for (SubTask st : subtasks) {
            Agent specialist = findSpecialist(st.getType());
            specialist.send(AgentMessage.task(st));
        }
        
        // 3. 收集结果
        List<String> results = collectResults();
        
        // 4. 整合输出
        return synthesize(results);
    }
}

// 专业Agent示例
public class SearchAgent implements Agent {
    public String process(String task, AgentContext context) {
        return searchTool.execute(task);
    }
}
```

## 七、结论与建议

### 7.1 核心结论

1. **PLAN模式相对最完整**，但Executor仍需增强为真实工具调用
2. **ASK/REACT无法真正工作**，核心原因是缺少Tool Registry
3. **AGENT_TEAM**仍为「同模型多段管线」而非分布式多 Agent；已具备多轮 Specialist+合成、专员可选工具与主/子职责提示词，与「真多专家」仍有距离
4. **CRON/STATE/HITL是MVP**，完全未达到设计要求

### 7.2 下一步建议

**立即执行（本周）**：
1. 梳理Tool Registry的需求和接口设计
2. 确定第一批需要实现的工具列表
3. 制定v0.4版本的开发计划

**短期目标（2-4周）**：
1. 完成Phase 1：Tool Registry
2. 基于Tool Registry重写ASK/REACT模式
3. 增强PLAN模式的Executor

**中期目标（1-2月）**：
1. 完成Phase 3-4：WORKFLOW增强 + AGENT_TEAM
2. 完成Phase 5-6：CRON调度 + STATE/HITL

### 7.3 风险提示

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 工具系统设计不当 | 高 | 参考LangChain4j的Tool接口 |

## 八、规范形态与实现差距（对照设计指南 §1.3.1）

设计指南已补充 [§1.3.1 核心执行与辅助及调度分类](./AI智能体行动模式设计指南.md#131-核心执行与辅助及调度分类) 及 **§8.0 / §9.0 / §10.0**、**§1.2.1** 等节，将**产品规范目标**与**当前仓库实现**对照。下列为便于排期与验收的**差距条目**（实现以 [开发路线图.md](../计划/开发路线图.md) 为准）。**B1 守门、流式分帧与 B2/B3/B4 极简草案**见 [v0.5-横切能力极简设计.md](./v0.5-横切能力极简设计.md)。

| 差距项 | 当前实现（摘要） | 规范目标（摘要） |
|--------|------------------|------------------|
| 显式 Plan / Agent Team 守门 | **已部分实现**：`IntentRouter` + `ModelIntentClassifier#classifyOptional`；降级时 `RoutingSource.EXPLICIT_GUARD`、非流式拼接说明、流式先 **BLOCK** 再 Ask **TOKEN**；分类失败则尊重显式 | 与左列「已实现」对齐；可选后续：配置化扩展 ReAct/Workflow |
| Cron 真调度 | `CronModeHandler` 演示文案；聊天 API 不开放 CRON | **库表**存任务 + **定时器**到期执行 + **双创建路径**（功能入口 / 模型工具）+ **全局/会话**作用域 |
| DST / STATE_TRACKING | 进程内**计数器** | 核心路径内**模型门控**发起追踪；结构化状态；**卡片**定稿；**侧栏**查看/改状态/确认并回注 |
| HITL | 独立 Handler 返回说明；无队列与续跑 | **嵌入**路径；**模型门控**发卡；暂停持久化；审批/拒绝后**同会话续跑**（协议另定） |
| Agent Team | Leader+多轮 Specialist+合成；专员在 `jingu3.tool.enabled` 时走 Ask 同款工具路由；提示词约束卡片确认/定时创建归主协调 | 并行专员、嵌套子团队、独立模型/记忆 per Agent |
| Workflow 与门控 | 未在本文单独要求 | 声明式人工节点与 LLM 节点门控**可并存**（见指南 §9.0 / §10.0） |

**相关代码锚点**：`ActionModePolicy`（对话可选五种核心模式）；`engine/mode/CronModeHandler.java`、`StateTrackingModeHandler.java`、`HumanInLoopModeHandler.java`；`AgentTeamModeHandler` + `AskModeHandler#runToolAugmentedOneShot`（专员工具）；意图路由可参考 `engine/routing/ModelIntentClassifier.java`（守门实现待路线图落地）。
