# 事件模型与有向关系类型

本文约定 **事件要素**（落 Elasticsearch 等）与 **事件 A → 事件 B 的有向关系**（落 Neo4j 等）。与 [记忆知识系统数据库选型分析第二版.md](./记忆知识系统数据库选型分析第二版.md)、[系统数据存储方案.md](./系统数据存储方案.md) 一致；ES 字段级映射见 [docs/data/elasticsearch/events-index.json](../data/elasticsearch/events-index.json)。

---

## 1. 事件要素（建议字段）

除 `event_id`、`conversation_id`（或 `session_id`）、`user_id`、`timestamp`、`action`、`result`、`metadata`、`vector_id`、`message_id` 等基础字段外，补充下列 **语义要素**（便于检索、过滤与下游抽取对齐）。

| 字段（ES 属性名） | 类型建议 | 说明 |
|-------------------|----------|------|
| `event_subject` | text + `.keyword` | **主体**：谁发起/施加该事件（施事、机构、角色等）；可存规范化短语 |
| `event_location` | text + `.keyword` | **地点**：事件发生的地点或空间范围 |
| `trigger_terms` | keyword（多值） | **触发词**：锚定事件类型的词（常为动词或名词）。例：「该公司**宣布**破产」→ `宣布`；「发生了一起车祸**事故**」→ `事故` |
| `modality` | keyword | **模态**：该片段在用户意图中是真实世界已发生/将发生，还是虚构、假设、玩笑、反事实等（取值见下表） |
| `temporal_semantic` | keyword | **时间语义**：事件在时间轴上的位置——过去 / 现在 / 计划未来 等（取值见下表） |

**`modality` 建议取值（机器码，可扩展）**

| 取值 | 含义 |
|------|------|
| `FACTUAL_ASSERTED` | 陈述为真实世界已发生或已存在 |
| `FACTUAL_FUTURE` | 真实世界中预期将发生（计划、预约） |
| `HYPOTHETICAL` | 假设、条件句中的非承诺性场景 |
| `FICTION` | 故事、角色扮演等非真实叙事 |
| `JOKE` | 玩笑、讽刺，不当作事实约束 |
| `COUNTERFACTUAL` | 反事实、与已知事实相反的假设 |
| `UNKNOWN` | 无法从文本判定 |

**`temporal_semantic` 建议取值**

| 取值 | 含义 |
|------|------|
| `PAST` | 已发生 |
| `PRESENT` | 当前正在进行或状态持续 |
| `PLANNED_FUTURE` | 已计划/排期的未来 |
| `TIMELESS` | 泛时陈述（规律、定义） |
| `UNKNOWN` | 无法判定 |

`actors` 可与 `event_subject` 并存：前者偏参与集合，后者偏**主施事**表述。

## 2. 有向事件关系（A → B）

所有入库关系均为 **从事件 A 指向事件 B** 的有向边。**「无关系」不存储**。

**机器码 `rel_kind`（写入 Neo4j 边属性，见第 3 节）与语义**

| `rel_kind` | 中文名 | A → B 的语义 |
|------------|--------|----------------|
| — | （无关系） | **不存储** |
| `CAUSATION` | 因果 | **A 导致 B**（因 → 果） |
| `EFFECT_CAUSE` | 果因 | **A 为果、B 为因**（与 `CAUSATION` 对偶：沿边从 A 读到 B 时，B 是因、A 是果） |
| `TEMPORAL_BEFORE` | 前后时序 | **A 早于 B**（时间先后） |
| `TEMPORAL_AFTER` | 后前时序 | **A 晚于 B**（与 `TEMPORAL_BEFORE` 对偶） |
| `CONDITION` | 条件 | **A 是 B 成立的前提**（条件 → 结果） |
| `CONDITION_INVERSE` | 逆条件 | **A 为结果侧、B 为条件侧**（与 `CONDITION` 对偶） |
| `PURPOSE_MEANS` | 目的 | **A 为达成 B 而发生**（手段/行动 → 目标） |
| `PURPOSE_GOAL` | 逆目的 | **A 为目标、B 为手段**（与 `PURPOSE_MEANS` 对偶） |
| `SUBEVENT` | 子事件 | **A 是 B 的子事件**（子 → 父） |
| `PARENT_EVENT` | 父事件 | **A 是 B 的父事件**（父 → 子，即 B 为 A 的子事件） |
| `OTHER_RELATION` | 其它关系 | 弱类型或待人工标注；**必须**提供 `explanation`（字符串） |

**约束**

- `OTHER_RELATION`：`explanation` 必填。
- 其余类型：`explanation` 可选（可用于模型置信度说明、引用 span 等）。
- 可选边属性：`confidence`（0～1）、`source`（`rule` / `llm` / `human`）、`created_at`。

## 3. Neo4j 建模建议

- **节点**：`:Event`，至少 `id`（= `event_id`）、`type`、`timestamp`；建议与 ES 对齐增加 `user_id`（与历史种子 `userId` 统一时做一次迁移）。
- **关系**：使用**单一关系类型** `EVENT_LINK`，避免 11 种类型在 Cypher 中爆炸式 `OR`；用属性区分语义：

```cypher
// 例：A 早于 B
MATCH (a:Event {id: $idA}), (b:Event {id: $idB})
CREATE (a)-[:EVENT_LINK {
  rel_kind: 'TEMPORAL_BEFORE',
  confidence: 1.0,
  explanation: null
}]->(b)

// 例：其它关系（explanation 必填）
CREATE (a)-[:EVENT_LINK {
  rel_kind: 'OTHER_RELATION',
  explanation: '用户自述与上文引用指向同一实体但无明确因果'
}]->(b)
```

**查询示例（因果闭包，仅 `CAUSATION`）**

```cypher
MATCH path = (start:Event {id: $eid})-[:EVENT_LINK*1..3 {rel_kind: 'CAUSATION'}]->(end:Event)
RETURN path
```

若需与仅支持多关系类型的旧图共存，可增加迁移：将 `CAUSED`/`PRECEDES` 等逐步收敛为 `EVENT_LINK` + `rel_kind`。

## 4. Milvus（事件向量，可选标量）

事件向量集合以 `event_id`、`user_id`、`ts_ms` 为主；若检索需按模态或时间语义过滤，可演进增加 `modality`、`temporal_semantic`（`VARCHAR`，与 ES 同码），**换字段需按 Milvus 迁移策略新建集合并回填**。
