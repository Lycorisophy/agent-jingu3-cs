# v0.6 接口文档 — 记忆实验 API

**基路径**：`/api/v1/memory`  
**统一响应**：`ApiResult<T>`（与全局约定一致）  
**开关**：`jingu3.memory.api-enabled`（默认 `true`）；为 `false` 时不暴露本组接口。

## 1. 创建记忆条目

- **POST** `/api/v1/memory/entries`
- **Content-Type**：`application/json`

**请求体**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| userId | string | 是 | 用户 ID；单用户阶段常用 `001` |
| kind | string | 是 | `FACT` 或 `EVENT` |
| summary | string | 否 | 摘要 |
| body | string | 否 | 正文 |
| factTag | string | 否 | 仅 `kind=FACT` 时写入 `fact_metadata.tag` |

**响应**：`ApiResult<MemoryEntryVo>`

**MemoryEntryVo 字段**：`id`、`userId`、`kind`、`summary`、`body`、`factTag`、`createdAt`、`updatedAt`（ISO-8601 字符串，以实际序列化为准）。

## 2. 按用户列出记忆条目

- **GET** `/api/v1/memory/entries?userId={userId}`
- **参数**：`userId` 必填

**响应**：`ApiResult<List<MemoryEntryVo>>`；按创建时间倒序，条数上限 `jingu3.memory.max-list-size`。

## 3. 更新记忆条目

- **PUT** `/api/v1/memory/entries/{id}`
- **请求体**：`userId`（必填，须与条目一致）；`summary`、`body`、`kind`、`factTag` **至少填一项**（与「全空」的 PATCH 区分）。
- **FACT 标签**：`factTag` 传入空字符串可清空 `fact_metadata`；不传 `factTag` 则不修改已有标签。
- **kind 改为 EVENT**：删除对应 `fact_metadata` 行。
- **响应**：`ApiResult<MemoryEntryVo>`；若启用 Milvus，会删除旧向量并重建（正文为空则仅删向量侧）。

## 4. 删除记忆条目

- **DELETE** `/api/v1/memory/entries/{id}?userId={userId}`
- **说明**：先删 Milvus 向量（若启用），再删 MySQL 行（`fact_metadata` / `memory_embedding` 随外键级联）。

## 5. OpenAPI（记忆）

仓库根 [docs/v0.5/openapi.yaml](../v0.5/openapi.yaml) 已包含 `/api/v1/memory/entries` 路径定义；以本文件与源码为补充说明。

## 6. Elasticsearch 事件 API（v0.6-C）

**前置**：`jingu3.elasticsearch.enabled=true` 且集群可达（默认 `http://127.0.0.1:9200`）。

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/events` | 请求体见 `CreateEventDocumentRequest`；未传 `eventId` 时生成 `evt_` 前缀 ID；返回 `ApiResult` 内含 `eventId` |
| GET | `/api/v1/events/search?userId=&q=&size=` | `q` 为检索词，`size` 默认 10、最大 50 |

**降级（依赖不可用）**：Elasticsearch 或索引异常时，`GET .../search` 仍返回 **HTTP 200** 与 **`data` 空数组**（服务端 WARN 日志）；`POST` 写入失败返回 **HTTP 503**，业务码 **`AG_50301`**（`DEPENDENCY_UNAVAILABLE`），提示事件未写入。

详见 [详细设计](./详细设计.md) 第 5.1 节与 [部署文档](./部署文档.md) 验收步骤。
