### 1.4 客户端 UI 与对话提示词可见性

- **前端 / 客户端 UI**：布局、组件与交互须对照上表《JinGu3 AI Agent 客户端 UI 设计规范》；与行动模式相关的控件命名、模式切换入口应与该文档及 [行动模式设计指南](../设计/AI智能体行动模式设计指南.md) 一致。
- **服务端提示词（ENG:提示词）**：无论行动模式来自**意图场景自动识别**（规则 / 模型分类）还是**用户显式选择**，凡调用 LLM 的拼装提示均须在段首注入**本轮路由说明**（选取方式 + 当前模式自然语言名）以及**各模式含义速查**，使模型能够善意回复例如「若需执行某类任务，请切换到某模式」。实现位置：`cn.lysoy.jingu3.prompt.ModeRoutingPreamble`、`PromptAssembly` 对各模式的组合方法；模式列表与释义维护在 `PromptTemplates#MODE_CATALOG_FOR_LLM`。

### 1.1 技术选型边界（栈与外集成）

- **Spring AI**：**不纳入**本仓库技术选型与新建依赖讨论。理由：其抽象范式与生态绑定对**企业级深度定制**（多模型路由、复杂编排、与既有域模型强融合、长期演进）约束偏多、**灵活性不足**；LLM 集成以当前及规划中的 **LangChain4j**（如 `langchain4j-ollama`）与自研引擎层为主。若未来有变更须**架构评审**并形成书面决议。
- **Flowable**：**纳入**技术选型**候选**。适用于需要 **BPMN 2.0**、人工任务（User Task）、流程实例与历史审计等与 **审批流、复杂工作流、HITL** 深度结合的场景；可与引擎侧 `ModePlanExecutor` / `WorkflowModeHandler` 等按模块边界对比选型（许可协议、运维成本、与 Spring Boot 集成方式在详细设计中评估）。

## 3. 每版本文档流水线与目录 `docs/{版本号}/`

### 3.1 规则

- 每个发布版本（如 `v0.1`、`v0.2`）的**过程文档**统一放在 **`docs/{版本号}/`**。
- **源代码**位于 `agent-jingu3-cs-server/`、客户端工程目录等，**不**迁入 `docs/`。
- 工序须**按顺序**完成；前一步未评审通过，不进入下一步。

### 3.2 工序顺序

1. **PRD**：须含**用户故事**、**验收标准**（可测、可追溯到路线图条目）；撰写方式对齐 SOP 阶段二「五步法 PRD」。
2. **可行性分析**：技术/依赖/风险/备选；对照本地 Ollama、单用户 `001` 等约束。
3. **概要设计**：边界、模块、与三大系统/三大工程映射、主接口与数据流。
4. **详细设计**：包/类、关键算法、错误处理、协议与存储级约定。
5. **服务端代码实现**：须含**必要注释**（类、公共 API、非显然逻辑）。
6. **接口文档**：REST/WebSocket 等契约（`接口文档.md` 或同目录 `openapi.yaml`）。
7. **前端代码实现**（若本版本范围包含 UI）。
8. **后端单元测试**：覆盖本版本服务端变更。
9. **部署文档**：构建、配置、启动、依赖（含 Ollama）、回滚要点。

### 3.3 建议文件名（复制到新版本目录时改名前缀路径即可）

- `PRD.md`
- `可行性分析.md`
- `概要设计.md`
- `详细设计.md`
- `接口文档.md`
- `部署文档.md`
- `README.md`（推荐：本版本文档与工序完成清单）

### 4.2 时间维度

### 4.3 与指南中分级的参考对照

| 本规范（时间） | 与指南中 tier/概念的关系（实现时声明） |
|----------------|----------------------------------------|

| 长期 | 对应 important 级向量检索、跨会话持久化等 |

## 12. Java 服务端编码约定（`agent-jingu3-cs-server`）

### 12.1 日志

- 生产代码使用 **Lombok `@Slf4j`**，通过 `log` 字段输出日志。
- 禁止手写 `LoggerFactory.getLogger`（单测类可保留手写或使用 `@Slf4j`，二选一即可）。

### 12.2 依赖注入

- **优先构造器注入**；禁止字段 `@Autowired`（测试类除外）。
- `@Configuration` 内 `@Bean` 方法通过参数列表注入依赖。

### 12.3 分包

| 类型 | 包路径（示例） | 说明 |
|------|----------------|------|
| 配置键等常量 | `cn.lysoy.jingu3.common.constant` | 与 `application.yml` 键名一致，如 `Jingu3ConfigKeys` |
| 跨模块枚举 | `cn.lysoy.jingu3.common.enums` | 如 `ErrorCode`；**领域枚举**（如行动模式）可保留在 `engine` 等域包 |
| 入参 DTO | `cn.lysoy.jingu3.common.dto` | HTTP 请求体 |
| 出参 VO | `cn.lysoy.jingu3.common.vo` | HTTP 响应负载（如 `ChatVo`；编排场景含 `planSteps`） |
| 应用服务 | `cn.lysoy.jingu3.service` | 用例编排（如 `ChatService`）；**Controller 保持薄**，不写业务分支 |
| REST 控制器 | `cn.lysoy.jingu3.controller` | `@RestController`，如 `ChatController` |
| 全局异常 / 种子常量组件 | `cn.lysoy.jingu3.component` | `ApiExceptionHandler`、`UserConstants` 等 |
| 统一响应 / 异常 | `cn.lysoy.jingu3.common.api` / `common.exception` | `ApiResult`、`ServiceException` 等 |
| 提示词与拼装片段 | `cn.lysoy.jingu3.common.constant` | `PromptTemplates`（系统/固定指令）、`PromptFragments`（换行与「用户：」等标签）、`EngineMessages`（校验与 Stub 文案）、`RoutingRuleKeywords`（规则路由关键词）、`LogMessagePatterns`（日志占位模板） |
| 提示词拼装 Bean | `cn.lysoy.jingu3.prompt` | `PromptAssembly` 仅负责组装，**不**内联长字符串 |

### 12.4 统一响应与异常

- 对外 REST 返回 **`ApiResult<T>`**：`success`、`code`、`message`、`data`、`timestamp`；成功时 `code` 为 `0`。
- 业务错误码见 **`ErrorCode` 枚举**（如 `AG_40001`）；抛出 **`ServiceException`**，由 **`@RestControllerAdvice`** 统一转换为 `ApiResult` 并设置 HTTP 状态。
- 校验失败、非法参数、未处理异常均在全局处理器中映射，**不**向客户端暴露堆栈细节。

### 12.5 配置注入（`@Value`）

- 所有 **`@Value("${...}")` 必须提供默认值**，与 `application.yml` 中约定一致，避免缺省配置导致启动失败。
- 示例：`@Value("${jingu3.user.id:001}")`、`@Value("${jingu3.user.name:user}")`、`@Value("${jingu3.ollama.base-url:http://localhost:11434}")`。
- 默认值中若含 **冒号**（如 Ollama 镜像标签 `...:35b-a3b`），在占位符内用 **`\:`** 转义，例如：`@Value("${jingu3.ollama.chat-model:gnremy/qwen3.5-abliterated\\:35b-a3b}")`（源码中为 `\\:`）。

### 12.6 DTO/VO 与行动模式（对话层）

- **DTO/VO**：使用 **Lombok + JavaBean**（如 `@Getter`/`@Setter`），**不使用** Java `record`；`ApiResult` 等同理。
- **对话可选模式**（`mode` / `modePlan`）：**ASK、REACT、PLAN_AND_EXECUTE、WORKFLOW、AGENT_TEAM**。逻辑见 `engine.ActionModePolicy`。
- **不得**在对话请求中直接选取：**CRON**（定时任务由独立模块或技能创建）、**STATE_TRACKING**、**HUMAN_IN_LOOP**（后两者为其它模式的辅助能力，由引擎/技能侧使用）。
- 选用 **WORKFLOW** 时建议传 **`workflowId`**；**未传**时服务端**不报错**，将本轮（或 `modePlan` 中该步）**回落为 ASK**。显式 `mode` 无法解析、或模型意图分类不可用时亦**回落为 ASK**（原 `IntentRouter` / `ModelIntentClassifier` 中部分场景曾为 REACT）。实现见 `engine.routing.RoutingFallbacks`。
