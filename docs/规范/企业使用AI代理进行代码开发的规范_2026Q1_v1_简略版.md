# 企业使用 AI 代理进行代码开发的规范

**2026Q1 v1 (简略版)**

---

## 文档使用说明

本《简略版规范》旨在为研发团队提供一套"10 分钟可读、即刻可执行"的 AI 开发红线与行为准则。它浓缩了详细版中 4 万字的核心治理逻辑，删减了复杂的背景论证，保留了必须执行的动作清单。请注意，简略版不代表标准的降低，而是对执行效率的极致追求。

### 适用范围与边界

本规范强制适用于**公司全体服务端研发人员**（包括后端、架构、中间件、平台工程及测试开发团队）。适用场景覆盖软件开发全生命周期（SDLC），具体包含但不限于：需求分析、架构设计、代码编写、单元测试、Code Review 以及运维脚本编写。

**工具覆盖范围：** 本规范同等约束"基于 Chat 的辅助编程"（如 Copilot Ghost Text, Chat Panel）与"基于 Agent 的自主代理开发"（如 Cursor Composer, Windsurf, Devin 等）。任何涉及将代码或业务逻辑委托给 AI 处理的行为，均受本规范管辖。

> **注：** 本规范暂不适用于市场营销文案、行政通知等非工程类 AIGC 内容的生成。

### 执行力度定义

为确保执行过程无歧义，我们沿用 RFC 2119 标准定义以下三个执行层级。请所有 Tech Lead 在 Review 时严格把关：

| 层级 | 定义 | 说明 |
|------|------|------|
| **MUST / 严禁** | 绝对的强制性约束 | 例如"严禁输入密钥"、"必须通过 CI 门禁"。违反此类条款等同于一级安全违规，将直接触发合规问责与绩效考核。此类条款没有任何"商量余地"，除非获得 CTO 级别的书面豁免。 |
| **SHOULD / 建议** | 强烈推荐的最佳实践 | 例如"建议使用 TDD 模式"、"应优先使用 SpecKit"。在大多数情况下你应当遵守，但如果在特定紧急场景（如线上 P0 级故障修复）下存在技术阻碍，可以经过 Tech Lead 评估后进行例外处理，但事后必须补齐相应文档或测试。 |
| **OPTIONAL / 可选** | 允许团队根据自身情况灵活选择 | 例如：探索性 Spike/原型验证场景可采用 Vibe Coding，但仍需遵守数据红线与最小门禁。此类条款赋予了一线团队自主权，但在选择时必须自行承担相应的风险（如代码返工率上升），并确保不突破上述 MUST 类红线。 |

### 版本与反馈

AI 技术演进极快，本规范实行**"季度（Quarterly）迭代制"**。当前版本为 2026Q1_v1，有效期至 2026 年 3 月 31 日。

如果在执行过程中发现规范阻碍了合理的生产力，或者发现了新的 Prompt 注入风险，请立即通过 [内部工单-AI 治理专区] 提交反馈。我们的原则是"治理服务于效率"，优秀的反馈将被采纳并记入技术贡献。

## 总纲：效率与治理的辩证统一

> **核心总纲：真正的效率提升，必须且只能在系统性治理的框架内发生。**

我们必须清醒地认识到，脱离了质量约束的"AI 编程速度"只是在加速技术债务的堆积。Gartner 预测 2028 年 75% 的工程师将使用 AI，但只有那些建立了完整治理体系的企业，才能将这种"代码生成能力"转化为可持续的"业务交付能力"。

### 五条核心带走点 (Key Takeaways)

如果你记不住整本规范，请务必死守以下五条底线，它们是保障你职业安全与系统稳定的生命线：

1. **数据红线不可越：** 数据分类分级（L1-L4）是物理隔离的铁律。核心密钥（AK/SK）、用户隐私（PII）与未公开财报绝对严禁输入公有云 AI 模型。无论 AI 多么智能，它本质上是一个不可控的第三方服务，泄密即事故。

2. **流程锁死不发散：** 严格执行 `Spec -> Plan -> Task` 的标准作业程序。坚决摒弃"一句话需求直接生成代码"的盲盒模式。必须先用文档定义清楚"做什么（Spec）"和"怎么做（Plan）"，再让 AI 去"做（Task）"。

3. **策略选择要分级：** 根据任务风险动态切换模式。对于常规 CRUD 迭代，默认使用 `Vibe + Rules` 提升手感；但对于涉及资金、鉴权的核心高危模块，必须强制切换到 `SpecKit + TDD` 模式，用测试用例作为硬性验收标准。

4. **CI 门禁是底线：** 自动化流水线（CI Pipeline）是最后一道防线。构建失败、SAST 高危漏洞、单元测试覆盖率不达标（按风险分级），一律 **Block**。严禁任何形式的"强制通过"或"人情放行"，AI 生成的代码必须比人工代码经受更严苛的机检。

5. **审计留痕防甩锅：** 人是最终责任人，但审计记录是还原真相的关键。所有 AI 生成的代码提交必须打上 `[AI]` 标记，MR 必须附带 Prompt/Spec 摘要。这不仅是为了追责，更是为了在发生"模型幻觉漂移"时，我们能复盘并优化 Rules。

## 六大原则：红线与默认设置

原则不是挂在墙上的口号，而是指导每一个工程决策的底层逻辑。我们将详细版中的六大原则提炼为以下执行导向的解释。

### 基础原则

| 原则 | 工程落地 |
|------|----------|
| **1. 质量优先 (Quality First)** | 效率提升不能以牺牲可维护性为代价。如果 AI 生成的代码虽然能跑通，但逻辑晦涩、缺乏注释或引入了新的技术栈（如莫名其妙引入了 lombok），必须在 Code Review 阶段无情打回。代码是给人看的，不是给机器存的。 |
| **2. 安全可控 (Security Controllable)** | 安全是 0，效率是 1。在任何时候，数据泄露风险都是一票否决项。如果不确定一段代码是否安全（例如复杂的正则或加密实现），必须人工逐行审查，或者直接弃用 AI 生成版本。 |
| **3. 人机协同 (Human-in-the-Loop)** | 禁止 "Auto-Commit" 和 "Auto-Deploy"。在 AI 生成任何实质性变更后，必须有人类工程师进行显式的"确认（Confirm）"或"审查（Review）"动作。人必须始终握有"刹车权"。 |
| **4. 持续演进 (Continuous Evolution)** | 今天的 Best Practice 明天可能就是 Anti-Pattern。团队应保持开放心态，通过 Champion Network 分享新发现的 Prompt 技巧，并定期更新 `.cursor/rules` 规则库。 |
| **5. 可追溯 (Auditability)** | 凡走过必留痕。我们不记录每一次 Chat 的闲聊，但必须记录每一次产生代码变更的"意图（Prompt/Spec）"和"结果（Commit）"。这是事故复盘的唯二依据。 |

### 原则 6：治理优先 (Governance by Design)

我们将"治理"硬编码进开发环境的三项默认设置：

1. **默认最小权限 (Default Least Privilege)**
   - **解释：** AI Agent 不应获得整个文件系统的读写权。默认情况下，Agent 只能读取当前上下文（Context）中的文件，且只能修改用户明确指定的文件。IDE 插件配置应默认关闭"自动读取全库"功能，防止无关敏感文件被上传。

2. **默认门禁阻断 (Default Block)**
   - **解释：** 在 CI 流水线中，针对 AI 生成代码的检查策略应设为"Strict Mode"。任何 Warning 级别的静态扫描问题（如未使用的变量、潜在的空指针），在 AI 代码中应升级为 Error 并阻断合并。我们不接受 AI 带来的"坏味道"。

3. **默认记录留存 (Default Log)**
   - **解释：** AI 网关（AI Gateway）必须默认开启日志记录功能。每一条发往 LLM 的 Request 和 Response 都必须被脱敏存储至少 6 个月。如果开发者试图绕过网关直接调用 API，网络层应默认阻断。

## 两套分级体系：工具 L0-L4 与 治理 L0-L3

> **注意：** 本文同时存在两套"L"编号——自动化分级 L0-L4（工具能力维度）与治理分级 L0-L3（组织维度）。当两者同句出现时，必须写成"工具 L2/治理 L2"以避免误读。

为避免概念混淆，我们明确区分"工具的能力等级（Automation Level）"与"管理的治理等级（Governance Level）"。这是两个独立的坐标系，请勿混为一谈。

### L0-L4：工具自动化分级 (Automation Levels)

此分级描述的是**"AI 能帮我做什么"**。级别越高，AI 接管的操作越多，风险也随之指数级上升。

| 等级 | 定义 | 典型工具形态 | 允许的权限范围 |
|------|------|--------------|----------------|
| **L0** | 无自动化 (Manual) | 记事本、纯 IDE | 纯人工操作，不涉及 AI 数据传输。 |
| **L1** | 辅助补全 (Completion) | Copilot Ghost Text (Tab 补全) | 仅允许读取当前文件上下文，建议单行代码，人工需逐行确认。 |
| **L2** | 会话辅助 (Chat) | Chat Panel (侧边栏问答) | 允许读取本地 codebase 进行解释/生成文档，但不直接修改代码文件。 |
| **L3** | 任务代理 (Agent) | Cursor Composer, Windsurf | 允许跨文件读写、运行本地测试。必须在人工"Approve"后方可写入磁盘。 |
| **L4** | 自主代理 (Autonomous) | Devin, GitHub Workspace | 允许自主提交 MR、修复 CI 错误。<span style="color: rgb(220, 38, 38)">**严禁在核心系统中使用。**</span> |

### 治理 L0-L3：治理分级 (Governance Levels)

此分级描述的是**"我必须遵守什么规矩"**。级别越高，适用的对象范围越小，但管控力度越严。

治理分级体系是本规范的核心骨架。我们通过建立 L0-L3 四级治理模型，旨在解决"一管就死，一放就乱"的难题。L0/L1 级侧重于基础底线与效率赋能，允许团队在红线之上灵活探索；而 L2/L3 级则侧重于确定性与安全性，通过刚性制度与自动化门禁，确保核心资产万无一失。

| 等级 | 名称 | 适用对象 | 必须产出物 & 门禁 |
|------|------|----------|-------------------|
| **治理 L0** | 全员通用 (Baseline) | 全公司研发人员（含后端/前端/测试/运维） | - **工具白名单：** 仅限使用通过安全评估的企业版工具。<br>- **数据红线：** 严格遵守 L1-L4 数据分类分级标准。<br>- **合规声明：** 签署《AI 辅助开发合规承诺书》。 |
| **治理 L1** | 团队规范 (Team) | 业务线研发团队 (Business Units / Squads) | - **统一 Rules：** 在项目根目录维护 `/.cursorrules`。<br>- **代码风格：** 统一 Lint/Format 配置。<br>- **提示词模板：** 建立团队共享的 Prompt 库。 |
| **治理 L2** | 项目约束 (Project) | 具体应用/微服务 (Microservices / Libraries) | - **Constitution：** 定义 `constitution.md` (架构边界)。<br>- **Spec 文档：** 必须产出 `spec.md` 并通过评审。<br>- **接口契约：** 明确 Swagger/Proto 定义。 |
| **治理 L3** | 系统红线 (Critical) | 核心交易/支付/鉴权 (Payment / Auth / Core) | - **强制 TDD：** 先写测试 (Test-First)，测试失败严禁提交。<br>- **SpecKit 完整模式：** 严格执行 Spec-Plan-Task 流程。<br>- **双人 CR：** 由 Tech Lead 或资深专家进行 Review。<br>- **强门禁：** 覆盖率 ≥ 90%，SAST 0 高危漏洞。 |

> **特别提醒：** 同一段落如同时提到工具等级与治理等级，统一写法"工具 Lx/治理 Ly"（例如：工具 L2/治理 L3），以避免概念混淆。

### 如何组合使用？

> **治理分级与自动化分级是两套独立坐标。** 不要以为治理 L3 级别的核心项目就不能用 AI。相反，你完全可以在治理 L3 严管项目中使用工具 L2（问答）或工具 L3（代理）工具，前提是你必须严格遵守治理 L3 的治理要求（TDD + 双人复核）。

**示例：** 在一个治理 L2（普通项目）中，可以使用工具 L3（Cursor Agent）进行开发；但在一个治理 L3（支付网关）中，可能仅允许使用工具 L1（补全）或受限的工具 L2，且必须先写测试。

## 策略怎么选：研发团队速查矩阵

面对不同类型的开发任务，Tech Lead 应在 5 分钟内根据下表完成策略决策。核心逻辑是：风险越低越追求速度（Vibe），风险越高越追求稳健（SpecKit）。

| 任务类型 | 典型场景 | 推荐策略 | 最小执行条件 (DoR/门禁) |
|----------|----------|----------|------------------------|
| **1. 探索性原型 / Spike** | 技术调研、一次性脚本、非生产 Demo | **Vibe Coding** (纯 AI 主导) | - 清晰的一句话目标<br>- 独立的代码分支/仓库<br>- 无需严格测试，允许代码抛弃 |
| **2. 常规功能迭代 / CRUD** | 后台管理接口、非核心报表、页面字段增删 | **Vibe + Rules** (默认推荐) | - 项目根目录含 `.cursor/rules`<br>- 明确的 API 定义 (Swagger/Proto)<br>- 通过基础 CI 扫描 |
| **3. 核心高风险** | 鉴权 (Auth)、支付、数据 Schema 变更、关键算法 | **SpecKit + TDD** (强制稳健) | - 必须撰写 `spec.md` 并通过人工 Review<br>- 必须先写测试用例 (Test-First)<br>- 双人 Code Review |
| **4. Bug Fix** | 线上问题修复、Sentry 报警处理 | **AI Analyze + TDD** | - 提供可复现的失败测试用例<br>- AI 分析根因 (RCA)<br>- 回归测试通过 |

### 决策原则

> **常规迭代默认 Vibe+Rules；核心高风险默认 SpecKit+TDD。** 如果无法判断风险等级，请遵循"就高不就低"原则，优先选择更严谨的策略。宁可开发慢一点，也不要事后救火。

## 最小可执行流程：Spec-Plan-Task

为防止 AI 在长任务中发散（Hallucination Loop），我们采用"阶段锁定"的标准作业程序（SOP）。本流程通过将大任务拆解为原子步骤，确保每一步都可验证。

### 标准作业流程 (SOP)

#### 1. 定义规格 (Spec Phase)

- **输入：** 原始需求 (PRD/User Story)、项目宪法 (Constitution)
- **动作：** 开发者与 AI 对话，明确 API 契约、数据结构、异常处理逻辑，生成 `spec.md`。
- **产出：** 清晰、无歧义的 Spec 文档。
- **Gate (硬关口)：** **Spec Review**。必须由 Tech Lead 或对等同事进行人工确认。Spec 不对，坚决不开工。

#### 2. 技术规划 (Plan Phase)

- **输入：** 确认后的 `spec.md`
- **动作：** AI 分析代码库，规划修改路径，生成步骤清单（ToDo List）。此时 AI **只读不写**，不修改业务代码。
- **产出：** `plan.md`（含依赖分析、影响面评估）。
- **Gate (硬关口)：** **Plan Confirmation**。开发者确认规划合理，无遗漏文件。

#### 3. 任务执行 (Task Phase)

- **输入：** 锁定的 `plan.md`
- **动作：** AI 逐个执行 Plan 中的原子任务（Implement）。每完成一个 Task，运行一次测试。
- **产出：** 代码变更 (Code Changes)、单元测试。
- **Gate (硬关口)：** **Test Passed**。单测绿灯。

#### 4. 审查验收 (Review Gate)

- **输入：** Merge Request
- **动作：** 人工 Code Review，CI 自动化扫描。
- **Gate (硬关口)：** **Security & Quality Check**。无高危漏洞，覆盖率达标。

### Plan/Develop 两段式执行规则

为避免 AI 越做越乱，必须严格区分两个模式：

- **Plan 阶段（思考模式）：** 禁止 AI 写代码。只让它看代码、想方案、写文档。如果发现 Spec 有漏洞，退回到 Spec 阶段修改，不要在 Plan 阶段打补丁。

- **Develop 阶段（执行模式）：** **锁定 Plan 文档。** 严禁 AI 在执行过程中随意发散（Scope Creep）或"顺便重构"。如果发现 Plan 行不通，必须停止执行，退回 Plan 阶段重新规划，严禁"硬写"。

## 三类资产：Rules / Constitution / Prompt

在 AI 时代，Prompt 和 Rules 是新的源代码。我们通过三层资产体系来管理 AI 的行为。

### 1. Rules (工程规范)

| 属性 | 说明 |
|------|------|
| **是什么** | AI 的"出厂设置"。定义代码风格、命名规范、技术栈版本。 |
| **放哪里** | `/.cursorrules` 或 `.cursor/rules/*.md` |
| **怎么改** | 视为"新源代码"。修改必须提 PR，由 Tech Lead 审核。 |

### 2. Constitution (项目宪法)

| 属性 | 说明 |
|------|------|
| **是什么** | 项目的"不可协商红线"。定义架构边界、安全约束、禁止使用的库。 |
| **维护人** | 仅限架构师 / Tech Lead |
| **最小模板** | 必须包含 Forbidden Libraries, Mandatory Patterns, Error Handling 策略。 |

### 3. Chain of Command

当指令冲突时，AI 听谁的？

1. Platform System Prompt (平台兜底)
2. Project Constitution (宪法)
3. User Prompt (你的指令)
4. Tool Output (工具结果)

> **规则：** 如果你的 Prompt 违反了 Constitution（例如要求关闭鉴权），AI 必须拒绝执行并报错。

## 质量门禁（按风险分级）

AI 生成代码容易出现"看起来对，实际有坑"的情况。我们不再实行一刀切的门禁，而是根据风险等级执行差异化阻断。

### Hard Gates (阻断性门禁)

| 门禁项 | 要求 |
|--------|------|
| **编译构建** | 默认建议 Warning 视同 Error（0 Warning），如历史项目存在大量遗留 Warning，可在治理 L2/L3 项目的 Constitution 中设定分阶段收敛目标。 |
| **安全扫描 (SAST)** | 默认建议 0 高危漏洞（High/Critical）作为阻断门禁；如遇误报或紧急修复场景，必须走 Tech Lead 审批并保留审计记录。 |
| **代码重复率** | 新增重复块 < 3%（防止 AI 简单复制粘贴）。 |

### 覆盖率分级阈值

**规则：** AI 生成代码量越大，测试要求越高。Tech Lead 需在 `constitution.md` 中固化本项目适用的具体阈值。

| 风险等级 | 增量覆盖率阈值 | 适用场景示例 |
|----------|----------------|--------------|
| **High Risk (高危)** | ≥ 90% | 支付、结算、鉴权、加密模块 |
| **Mid Risk (中等)** | ≥ 80% | 核心业务流程、订单流转、库存扣减 |
| **Low Risk (低危)** | 不低于项目基线 或 ≥ 70% | 后台查询接口、数据报表、辅助工具 |

### TDD + AI 五步法

对于治理 L3 级系统，必须严格执行：

1. 人写 Spec
2. AI 生成测试用例 (Test)
3. 人 Review 测试 (Gate)
4. AI 写实现代码 (Code)
5. CI 绿灯通过 (Pass)

## 安全红线（数据与供应链）

### 数据分级输入策略

严禁将高等级数据输入低等级信任域。

| 等级| 输入策略 | 典型红线示例 |
|------|----------|--------------|
| **L4 (绝密)** | **严禁输入** | 生产环境私钥 (AK/SK)、核心算法源码、未公开财报 |
| **L3 (机密)** | **需审批 + 脱敏** | 用户真实 PII (手机/身份证)、完整数据库 Dump |
| **L2 (内部)** | 允许输入企业版 AI | 内部 API 文档、非核心业务逻辑、脱敏后的日志 |
| **L1 (公开)** | 允许输入 | 开源框架代码、错误堆栈信息 (无 PII) |

### 供应链安全行动

| 行动 | 说明 |
|------|------|
| **密钥轮换 (Rotate)** | 假设历史代码库已泄露。对所有曾在 GitHub 托管过的密钥进行强制轮换。 |
| **Pre-commit 阻断** | 部署本地 Hook，物理拦截任何疑似 Secret 的字符串提交。 |
| **SBOM 管理** | AI 引入的新依赖包，必须纳入软件物料清单并进行 CVE 扫描。 |

### 疑似敏感信息处理 SOP

如果在 AI 对话框中不小心粘贴了敏感信息：

1. **停止 (Stop)：** 立即停止当前对话，不要试图解释。
2. **删除 (Delete)：** 删除该 Session/Chat 历史。
3. **上报 (Report)：** 向安全团队报备（免责条款：主动报备可免责）。
4. **轮换 (Rotate)：** 如果是密钥，立即作废并轮换。

## 审计留痕与事故响应

### 审计留痕三件套

| 序号 | 项目 | 说明 |
|------|------|------|
| **1** | **Commit 标记** | 提交信息必须包含 `[AI]` 或 `Co-authored-by: Copilot`。这是识别风险代码的基础。 |
| **2** | **MR 摘要** | 必须在 Merge Request 描述中附带 Spec 链接或 Prompt 摘要，证明代码意图。 |
| **3** | **模型版本** | 记录生成该代码的模型版本（如 `gpt-4o-20240513`）。<br>**目的：** 应对"模型行为漂移"，方便追溯。 |

### 事故响应 SOP (4 步法)

当发生疑似由 AI 代码导致的线上故障时：

1. **止损 (Mitigate)：** 优先回滚版本，切断流量。立即暂停该项目使用 AI 工具，防止 AI 基于错误上下文继续"胡说"。

2. **取证 (Evidence)：** 提取生成该代码的 **Prompt 记录、Spec 文档、模型版本**。不要只看代码，要看"它是怎么被生成出来的"。

3. **分析 (Analyze)：** 进行 5Why 分析。根因通常是：Spec 描述不清？Prompt 误导？还是 Reviewer 盲目 Accept？

4. **改进 (Improve)：** **必须将本次教训写入 .cursor/rules**，形成一条新的"防复发规则"（例如：禁止使用某个有 Bug 的库）。
