**史诗顺序（v2.0 起）**：① 意图场景识别 + 八大行动模式（v0.1～v0.2 已完成）→ **② 工具真能力 + 编排增强 + 横切能力（新 v0.3～v0.5）** → ③ 记忆系统（v0.6）→ ④ 技能系统扩展（v0.7）→ ⑤ 用户系统（v0.8）。

**与差距分析的关系**：报告结论认为 **Tool Registry** 为 P0 根因；故将「工具子系统 MVP」前移到 **v0.3**，原路线图中的「v0.3 记忆」整体后移至 **v0.6**，避免与指南 §3/§4 对齐冲突。详见各版本节内说明。

**文档约定**：每个版本的过程文档位于 **`docs/{版本号}/`**；工序与清单见 [docs/v0.1/README.md](../v0.1/README.md) 模板（各版本可复制结构）。**补充设计**（不限于某一 v0.x 目录）见 `docs/workflow/` 等子目录；其中 [workflow-design.md](../workflow/workflow-design.md) 为 **BPMN/Flowable 扩展愿景**，与下述 v0.4「JSON 工作流」里程碑区分。

**本地 Ollama**：主模型 `gnremy/qwen3.5-abliterated:35b-a3b`，嵌入 `qwen3-embedding:8b`。

**单用户（用户系统完成前）**：用户 ID `001`，用户名 `user`。

**客户端（独立仓库）**：每轮响应中的 **`actionMode` / `routingSource`**（非流式见 `ChatVo`，流式见首包 `StreamEvent` `META`）为「当前行动模式」的**唯一事实来源**；`routingSource` 含 **`EXPLICIT_GUARD`**（显式 Plan/Team 经守门降为 Ask，见 v0.5 进展）。UI 展示、编排逐步模式、`TOOL_RESULT` 等与后端对齐约定见 [JinGu3-AI-Agent客户端UI设计规范.md](../设计/JinGu3-AI-Agent客户端UI设计规范.md) **§1.4、§5.6**。

## v0.5 — Cron + HITL + State 持久化 + Agent Team 增强边界（差距整改 · 横切）

**版本状态**：进行中（2026-04-12 起）：**B1 显式模式守门**已代码落地；Cron / HITL / State 持久化等仍待开发。

**目标**（对齐差距分析 §3.5–3.8）：**Cron** 任务持久化与调度触发；**HITL** 审批队列与审批 API；**State** 任务级状态表（与 conversation 关联）；**Agent Team** 在可控范围内增强（多轮/多角色消息结构，见 `docs/v0.5/详细设计.md`）。与 [Cron与HITL模块设计草案.md](../设计/Cron与HITL模块设计草案.md) 对齐。

**进展（2026-04-12）**

- **守门（B1）**：`IntentRouter` 在显式 `PLAN_AND_EXECUTE` / `AGENT_TEAM` 下调用 `ModelIntentClassifier#classifyOptional`；分类为 ASK/REACT 时实际执行 **Ask**，`RoutingSource` 为 **`EXPLICIT_GUARD`**，`RoutingDecision#guardUserNotice` 为用户可见说明；分类器调用失败则尊重显式模式。非流式：`reply` 前缀拼接；流式：`META` 后先发 **`BLOCK`（说明）**，再进入 Ask `TOKEN` 流。详见 [v0.5-横切能力极简设计.md §1](../设计/v0.5-横切能力极简设计.md)。
- **设计草案**：同文档含阶段 C 决议（ReAct/Workflow 暂不同等守门；`cardKind`：`state_digest` / `action_approval`）及 B2/B3/B4 极简表字段与 REST 占位。
