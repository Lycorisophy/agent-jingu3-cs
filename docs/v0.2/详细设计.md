## 2. 请求与响应

### 2.1 `ChatRequest`

| 字段 | 类型 | 说明 |
|------|------|------|
| message | string | 必填 |
| mode | string | 可选，单模式显式 |
| conversationId | string | 可选 |
| modePlan | string[] | 可选；**非空时优先于 mode**；元素为 `ActionMode` 名，顺序即执行顺序 |

### 2.2 `ChatVo`

| 字段 | 类型 | 说明 |
|------|------|------|
| userId / username / reply | string | `reply` 为单模式或编排**最后一步**输出 |
| actionMode | string | 单模式时为该模式；编排时为**最后一步**模式名 |
| routingSource | string | 含 `FALLBACK`（非法单 mode 降级） |
| planSteps | array | 仅当请求含非空 `modePlan` 时非 null，元素为 `PlanStepVo` |

### 2.3 `PlanStepVo`

| 字段 | 类型 |
|------|------|
| mode | string |
| reply | string |

## 3. `ExecutionContext` 扩展

新增可选字段 **`taskPayload`**（可空）：本步交给 LLM 的有效载荷；为空则使用 **`userMessage`**。  
提供方法 **`llmInput()`**：`taskPayload` 非空且非 blank 则用之，否则 `userMessage`。  
**AskModeHandler / ReActModeHandler** 及所有新模式 handler 的 prompt 入参统一改为 **`context.llmInput()`**。

编排器步间拼接：`taskPayload = userMessage + "\n---\n上一步输出：\n" + previousReply`（分隔符常量放 `PromptFragments` 或 `EngineMessages`）。

## 4. 各模式 MVP 行为

| 模式 | 行为摘要 |
|------|-----------|
| ASK / REACT | 与 v0.1 一致，走 `llmInput()` |
| PLAN_AND_EXECUTE | 两次 generate：先「仅输出计划」，再「按计划与用户输入执行并回答」 |
| WORKFLOW | 两次 generate：固定「需求摘要」→「方案输出」 |
| AGENT_TEAM | 两次 generate：主 Agent 拆解 → 子 Agent 模拟执行 |
| CRON | 无 LLM 或可选一行；返回含默认 cron 表达式说明的文案（常量 + 配置默认值） |
| STATE_TRACKING | `ConcurrentHashMap<conversationId, 计数>` 递增并返回状态说明 |
| HUMAN_IN_LOOP | 返回「待人工确认」类固定文案（常量） |

## 5. 编排器

- 类：`ModePlanExecutor`  
- 入参：解析后的 `List<ActionMode>`（解析时非法项 → `REACT`）  
- 软上限：最多 **8** 步，超出截断并 `warn` 日志  
- 输出：`List<PlanStepVo>` + 最终 `reply` 文本  

## 6. 路由

- 单 `mode` 非法：`IntentRouter` 返回 `REACT` + `RoutingSource.FALLBACK` + `note`。  
- `modePlan` 存在：**不调用** `IntentRouter`；`routingSource` 记 **`CLIENT_EXPLICIT`**，`note`=`modePlan`。
