# v0.2 可行性分析

## 1. 技术可行性

- **顺序编排**：在单 JVM 内对 `List<String>` 解析为 `ActionMode`，循环调用既有 `ActionModeHandler`；步间用字符串拼接上一部输出作为 `taskPayload`，无需分布式编排。
- **八模式 MVP**：除 Ask/ReAct 外，六种模式采用「固定步数 LLM 调用 + 约定说明文案」或进程内 Map（状态/HITL/Cron）即可满足演示，不依赖新中间件。
- **非法 mode 降级**：在 `IntentRouter` 显式分支捕获解析异常，返回 `ASK` + `FALLBACK`（与 `ModelIntentClassifier` 不可解析时回落 `ASK` 一致）；`modePlan` 中单步非法 token 仍降级为 `REACT`，见 `ModePlanExecutor`。

## 2. 依赖与资源

- 仍依赖本地 Ollama（与 v0.1 相同）；Plan/Workflow/AgentTeam 等增加调用次数，耗时与 token 上升，可接受于 MVP。

## 3. 风险与缓解

| 风险 | 缓解 |
|------|------|
| LLM 输出不稳定 | 各模式 handler 内 try-catch，失败时返回约定错误文案常量 |
| modePlan 过长导致超时 | PRD 非目标不限长度；实现可对步数设软上限（如 8）并在详细设计中说明 |

## 4. 结论

可行；按详细设计实现编排器与各模式 handler，并补充单测与接口文档。
