# v0.2 接口文档

## 1. 概述

- **Base URL**：`http://localhost:8080`（默认端口）
- **内容类型**：`application/json`
- 统一响应体 **`ApiResult<T>`** 与 v0.1 相同；业务码见 `ErrorCode`。

## 2. 健康检查

同 v0.1：`GET /actuator/health`。

## 3. 聊天 `POST /api/v1/chat`

### 3.1 请求体 `ChatRequest`

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| message | string | 是 | 用户消息 |
| mode | string | 否 | 单模式显式枚举名；**若 `modePlan` 非空则忽略本字段** |
| conversationId | string | 否 | 会话 ID；缺省 `default` |
| modePlan | string[] | 否 | 多步模式名，**顺序执行**；**非空时优先于 `mode`**；单步无法解析为枚举名时该步降级为 `REACT`；最多 **8** 步，超出截断 |
| workflowId | string | 否 | 选用 **WORKFLOW** 时传入 classpath 工作流定义 id；**缺省时该步/本轮不执行工作流链，回落为 ASK**（同一请求内多步 WORKFLOW 共用该 ID） |

**对话可选模式**（`mode` / `modePlan` 中允许出现的枚举名）：**ASK、REACT、PLAN_AND_EXECUTE、WORKFLOW、AGENT_TEAM**。  
**不得**在 `mode` / `modePlan` 中选取：**CRON、STATE_TRACKING、HUMAN_IN_LOOP**（否则 **400** + `AG_40001`）。  
说明：定时任务由独立模块或技能创建；状态追踪与人在环为其它模式的辅助能力，不通过本接口直接选模式。

**规则路由**（无显式 `mode` 时）仅对关键词映射 **PLAN_AND_EXECUTE、WORKFLOW、AGENT_TEAM**；不再通过「定时/状态/人工」等关键词映射到上述三种不可选模式。

### 3.2 成功时 `data`：`ChatVo`

| 字段 | 类型 | 说明 |
|------|------|------|
| userId | string | 单用户种子 |
| username | string | 单用户种子 |
| reply | string | 单模式时为该模式输出；**编排时为最后一步输出** |
| actionMode | string | 单模式时为选用模式；编排时为**最后一步**模式名 |
| routingSource | string | `CLIENT_EXPLICIT` / `RULE` / `MODEL` / `FALLBACK`（显式 `mode` 无法解析、或模型分类不可用时等多为 **ASK** 回落，见实现） |
| planSteps | array \| null | 仅当请求含非空 `modePlan` 时返回 **非 null** 数组；元素见下 |

### 3.3 `planSteps[]`：`PlanStepVo`

| 字段 | 类型 |
|------|------|
| mode | string | 本步实际执行模式（含降级后的 `REACT`） |
| reply | string | 本步输出 |

### 3.4 示例：单模式（无法解析的 mode 回落 ASK）

```http
POST /api/v1/chat
Content-Type: application/json

{
  "message": "你好",
  "mode": "NOT_VALID"
}
```

成功响应（HTTP 200）示例：

```json
{
  "success": true,
  "code": "0",
  "message": "成功",
  "data": {
    "userId": "001",
    "username": "user",
    "reply": "...",
    "actionMode": "ASK",
    "routingSource": "FALLBACK",
    "planSteps": null
  },
  "timestamp": 1710000000000
}
```

### 3.5 示例：显式选取不可对话模式（失败）

```json
{
  "message": "test",
  "mode": "CRON"
}
```

HTTP **400**，`code` 为 `AG_40001`，`message` 说明该模式不可通过对话接口选取。

### 3.6 示例：WORKFLOW 与 workflowId

```json
{
  "message": "执行工作流",
  "mode": "WORKFLOW",
  "workflowId": "demo-wf-1"
}
```

若选用 `WORKFLOW` 但未提供 `workflowId`，服务端**不报错**，将本轮路由**回落为 ASK**（单轮问答）；`modePlan` 中含 `WORKFLOW` 步且无 `workflowId` 时，该步同样按 ASK 执行。

### 3.7 示例：编排 `modePlan`

```json
{
  "message": "分析并总结",
  "modePlan": ["ASK", "REACT"]
}
```

`planSteps` 为两步，`reply` 为第二步输出。
