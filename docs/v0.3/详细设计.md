# v0.3 详细设计 — 工具与 Ask / ReAct 集成

## 1. 对照指南

| 指南章节 | 本设计落点 |
|----------|------------|
| §3 Ask | 工具选择器 → 工具执行 → 结果汇总；未命中工具时保持单次 LLM |
| §4 ReAct | Thought → Action（工具调用）→ Observation（工具输出）→ 循环直至终止条件 |

## 2. 架构概览

```
                    ┌─────────────────┐
                    │ ToolRegistry    │
                    │ (id → Tool)     │
                    └────────┬────────┘
                             │
     AskModeHandler ─────────┼──────── ReActModeHandler
                             │
                    ┌────────▼────────┐
                    │ ToolExecutor    │
                    │ execute(name,   │
                    │   args)         │
                    └────────┬────────┘
                             │
                    ┌────────▼────────┐
                    │ Built-in tools  │
                    └─────────────────┘
```

- **实现选型**：优先使用 LangChain4j 已有 `Tool` / 适配层，与现有 `ChatLanguageModel`、`StreamingChatLanguageModel` 并存；若需统一抽象，在本包 `cn.lysoy.jingu3.tool`（示例包名，实现时以代码为准）定义 `JinguTool` 再适配到 LangChain4j。

## 3. 核心组件

### 3.1 ToolRegistry

- 启动时注册内置工具列表；支持按名称查找。
- 每个工具包含：`name`、`description`（进入 LLM 系统提示或 tool 列表）、参数 JSON Schema（最小子集：string/number/integer）。

### 3.2 工具调用与 Ask

**方案 A（推荐）**：利用支持 tool-calling 的 chat API（若 Ollama 模型与 LangChain4j 组合支持）：单次或少数轮次返回 `tool_calls`，执行后合并为最终答复。

**方案 B（兼容）**：两轮提示——第一轮仅输出 JSON `{ "tool": "...", "arguments": {...} }` 或 `NO_TOOL`；第二轮将工具结果与用户问题拼入 prompt 生成答案。v0.3 若 A 受阻可采用 B，并在 PRD「已知限制」中说明。

### 3.3 ReAct 真循环

- 每步：`buildReactLoopStepPrompt` 增强为携带 **上一工具输出**（Observation），而非仅模型自写观察。
- 终止条件：保持 `[TASK_COMPLETE]` 或改为「无工具可调用 + 模型声明完成」二选一（详细实现定稿时锁定一种）。

### 3.4 流式（可选增量）

- `ChatStreamService` 在工具执行前后可发送 `StreamEvent` 类型 `BLOCK` 或新增 `TOOL_CALL`/`TOOL_RESULT`（若扩展 `StreamEventType`，需同步前端约定）。本版本可先 **BLOCK** 整段工具结果。

## 4. 内置工具示例

| 工具 ID | 行为 | 用途 |
|---------|------|------|
| `calculator` | 安全解析四则运算表达式 | 验证数值类 Ask |
| `echo_fact` | 返回固定知识片段或回显参数 | 验证链路与日志 |

## 5. 错误处理

- 未知工具名：返回结构化错误，ReAct 将该错误作为 Observation 进入下一步或终止。
- 参数校验失败：同上。

## 6. 测试策略

- `ToolExecutor` 单测：合法/非法参数。
- Handler 集成测试（可 Mock LLM）：固定返回触发工具调用的文本，断言执行路径。
