# v0.3 PRD — 工具子系统 MVP + Ask / ReAct 真闭环

## 1. 背景与目标

依据 [行动模式实现差距分析.md](../设计/行动模式实现差距分析.md)，当前 **ASK** 仅为单次 LLM 调用，**REACT** 无真实 Action/Observation。本版本目标是落地 **Tool Registry（或等价薄封装）** 与执行管线，使 Ask、ReAct 符合《AI智能体行动模式设计指南》§3、§4 中对「工具调用与观察回流」的基本要求。

## 2. 用户故事

1. **作为** 终端用户 **我希望** 在 Ask 模式下当问题需要计算或结构化事实时 **系统能** 调用内置工具并汇总结果 **以便** 获得可核验的答案。
2. **作为** 终端用户 **我希望** ReAct 循环中「行动」对应真实工具执行、「观察」为工具输出 **以便** 行为可追溯、非纯文本扮演。
3. **作为** 运维/开发 **我希望** 工具调用有统一注册与日志 **以便** 扩展新工具与排障。

## 3. 范围

### 3.1 包含

- 工具注册表、工具元数据（名称、描述、参数 Schema 子集）、同步执行接口。
- 至少 **2** 个内置工具（示例：简单数学计算、回显/占位「检索」）。
- `AskModeHandler`：在「直答」与「需工具」间可路由（规则或单次 LLM 决策，详细设计定稿）。
- `ReActModeHandler`：每步生成可解析的「工具调用意图」→ 执行工具 → 将结果写入下一轮 prompt 作为 Observation。
- 单元测试覆盖工具执行与 handler 主路径。

### 3.2 不包含（后续版本）

- 客户端工具、流式工具输出的完整产品化（可与 [ChatStreamService](../../agent-jingu3-cs-server/src/main/java/cn/lysoy/jingu3/service/ChatStreamService.java) 增量约定）。
- 记忆系统（顺延至 v0.6）。
- 用户系统与多租户。

## 4. 验收标准

| 编号 | 验收项 | 验证方式 |
|------|--------|----------|
| AC-1 | Ask：输入触发工具路径时，响应中可见工具产出的实质内容（非纯模型臆造占位） | 手工或集成测试用例 |
| AC-2 | ReAct：至少一轮循环中 Observation 文本与工具返回一致 | 断言或日志比对 |
| AC-3 | 新工具可通过注册接口加入并被调用 | 文档化扩展步骤 + 测试双工具 |
| AC-4 | 工具执行异常有明确错误信息，不静默失败 | 单测或场景测试 |

## 5. 风险与依赖

- **Ollama + LangChain4j**：工具调用形态需与所选 API（如 `ToolSpecification`）对齐，详见 `详细设计.md`。
- **安全**：本版本仅内置低危工具；高危能力留待 v0.7 与 HITL 协同。
